## 待决策(Q1 ~ Q5)

### Q1 · 裁决"决议形态"

**A**:每个 dissent 给召集人 4 选 1 + 必填理由 文本框:
- `pick_a` —— 采纳 involved_agents[0] 的立场
- `pick_b` —— 采纳 involved_agents[1] 的立场
- `compromise` —— 自己写折衷决议(必填)
- `defer` —— 不裁决,记录"待后续讨论"(下次会议带回来)

review_decision 字段存 JSON `{"action": "pick_a|pick_b|compromise|defer", "rationale": "..."}`

**B**:不区分选项,只一个文本框,让召集人自己写决议。最简但失去结构。

**C**:除了 A 的 4 个之外再加 `restart_agenda` —— 全部重跑该议程(orchestrator 单议程 replay)。

> **我的推荐**:**A**。结构化但不臃肿;`defer` 留出口;`restart_agenda` 太复杂(要解决:重跑期间 老 consensus 怎么处理? 删 vs 软删? 老 action_items 怎么处理?),建议留到 v26.3.1 再做。

### Q2 · 沉淀范围

裁决完成后 把决议沉淀回 哪些 AI 专家 KB?

**A**:`involved_agents` 全部(分歧涉及的专家都要更新认知,不论裁决偏向谁)
**B**:只沉淀给被采纳方(被否的不沉淀,免得给"错误立场"加权)
**C**:沉淀给本议程涉及的所有 expert(包括没在 dissent 里的旁观者,广播性)

> **我的推荐**:**A**。被否方也需要知道"这个分歧上召集人选了对方,理由是X",下次类似议题它要顺着这个调子;但不广播给旁观者,避免噪音。

### Q3 · 沉淀的内容形态

沉淀什么文本?

**A**:LLM 二次生成 4 段(背景 / 双方立场 / 召集人裁决 / 关键教训),类似 v26.2 task consolidation
**B**:模板化拼字符串:`议程:<title>\n分歧点:<point>\n双方立场:<summary>\n召集人裁决:<rationale>(选了 <action>)`,不走 LLM
**C**:**A** 但 LLM 失败时 fallback **B**

> **我的推荐**:**C**。质量优先 + 兜底。LLM 调用约 1~2 秒。

### Q4 · 裁决后是否重跑 action_extractor?

后果:已抽出的 action items 若基于"未裁决前"的共识,可能跟决议矛盾。

**A**:**重跑**,删 旧 auto-extracted action items + 用新 summary_md(含决议)重抽。风险:可能误删用户手改过的 task
**B**:**不重跑**,让召集人 / leader 手工管理 task
**C**:**软合并**:只补 新 task,不删旧的。LLM 比较新旧 summary diff

> **我的推荐**:**B**。重跑风险高(删用户手改过的 task 是灾难);裁决产生的"新 task"应该让召集人在 UI 上手动补,这才是召集人岗位的本职动作。沉淀回 KB 已经让 AI 下次自己改了。

### Q5 · 裁决权限

**A**:只有 meeting.organizer_user_id(召集人本人)
**B**:owner/admin/leader 也可裁决
**C**:开放给所有 attendee_user_ids

> **我的推荐**:**A**。"召集人"在产品定义里是裁决责任人;若召集人离职 owner 可在 admin 后台 reassign organizer。不开放给所有 attendee — 否则会出现"5 个人 5 个裁决"的奇景。

# P0 #3 · 45 分钟整场硬上限(Q2 决策落地)

## 现状盘点

已有:
- `MAX_TURNS_PER_AGENDA = 6` 在 orchestrator
- orchestrator 主循环 `for idx, item in enumerate(agenda):` 顺序跑议程,无总时长检查

- 每议程 elapsed_sec 写入 MeetingConsensus(成本审计用)

## 待决策(Q6 ~ Q9)

### Q6 · 超时后是 FAIL 还是 COMPLETE?

**A**:`FAIL` — 标失败,last_error 写 "meeting_timeout",已跑完议程的 consensus 保留,**不**触发 summary + action_extractor
**B**:`COMPLETE` — 标成功,但 summary 里加"⚠️ 因 45 分钟超时,后 N 议程未跑",已跑完议程的 consensus 保留并触发 finalize 链路
**C**:**软 timeout** — 第 1 次到 45 分钟时 暂停(切 PAUSED),等召集人决定 续 30 分钟 还是 强制 finalize

> **我的推荐**:**B**。`FAIL` 会让用户觉得"白开了一场",但其实前几议程的 consensus + summary 是有价值的;客户不接受"45 分钟整场作废"。`C` 太交互复杂(召集人可能不在线)。**B** 是务实选项:把跑完的议程价值落地 + 在 summary 里明确告知有几条议程没跑。

### Q7 · 硬上限可配吗?

**A**:全局常量 `MAX_MEETING_SECONDS = 45 * 60`,代码 hardcoded
**B**:`Meeting.max_total_seconds` 字段,创建会议时可填(默认 2700)
**C**:workspace 级 settings.max_meeting_seconds + Meeting 级 override

> **我的推荐**:**A** 先做。客户当前无 per-meeting 调整需求;Meeting 字段加了就要 migrate + schemas + UI 三处改;留 v26.3.1 再升级到 B/C。

### Q8 · paused 时间算不算?

**A**:**算**(用 phase=running 起的 wall clock)
**B**:**不算**(用 running 累计时长,paused 时计时器停)

> **我的推荐**:**B**。召集人暂停 50 分钟去开会,回来后会议被强 finalize 显然不合理。auto_state 加 `running_started_at` + `paused_running_seconds` 两个字段实现。

### Q9 · 前端是否显示"剩余 / 已用"时间?

**A**:**显示** — 控制台 phase 徽章 旁加"剩余 33:20",到最后 5 分钟变红
**B**:**不显示** — 后端硬卡就行,UI 安静
**C**:**只显示 已用** — 不显示 上限,避免给客户"为什么要硬上限"的解释成本

> **我的推荐**:**A**。透明化对召集人有用(可以选择"还有 5 分钟就主动暂停跳过这议程");30 分钟内灰色,30–40 黄,40+ 红色提示。
