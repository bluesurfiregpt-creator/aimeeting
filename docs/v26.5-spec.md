## 二、候选池(按 v26.4 spec 留的项 + 演示收集反馈)

### C1 · 跨 workspace 批量推送 agent / KB 模板 ★(乙方运营痛点)

**背景**:你 维护 N 家客户,每家 workspace 都有 16 个 AI 专家 + 各种 KB 文档.如果 总部 升级了 一个 prompt 模板(eg 加了一段反幻觉规则),现状 必须 一个 workspace 一个 workspace 手动同步 — N 个客户 N 次重复劳动.

**设计点**:
- "**主 workspace**" 概念 — 平台超管的 自己 workspace 作为 template source
- **agent_template** 表:跨 workspace 的 agent 配置基线(name / description / role / system_prompt 等)
- "**推送同步**" 操作:平台超管在 /super 选 N 个 workspace,选 推什么(eg "把 主 workspace 的 AI-08 物业监管 system_prompt 推到所有客户的对应 agent")
- **diff 预览**:推之前 给 一份 N 个 workspace 各自当前值 vs 主版本 的 diff,**人工确认**(避免误推)
- **回滚**:推完后 24h 内可 一键 roll back

**风险**:
- 客户 自己改过 自己 workspace 的 agent prompt — 强推 会覆盖客户改动
- 解决:diff 预览 + 客户改过的 标 ⚠️ + 推送有 "保留客户改动 / 强推" 选项

**工时**:~2 天

---

### C2 · workspace suspend / archive / delete(write-heavy)

**背景**:v26.4 spec Q5 留的 write-heavy 三件套.

**设计点**:
- **suspend**:`Workspace.status='suspended'` — 该 ws 所有写端点 403,read 仍可(让 客户 仍能看 但 不能动);超管点一下即可恢复
- **archive**:`status='archived'` — 列表默认隐藏(`include_archived=true` 才显示);read 也封,只 超管能看
- **delete**:**软删** 标记 deleted_at(不真删),保留 ~90 天后自动 hard delete cron;hard delete 顺手 OSS 资料一并清

**触发场景**:客户合同到期 / 客户离职 owner 不交接 / 客户重复创建空间想清掉

**工时**:~1 天

---

### C3 · 计费 / 配额 / LLM 用量 dashboard

**背景**:对外授权变多以后,你 不知道 哪个客户 烧 多少 钱.而且 客户也想知道 自己 用了多少 LLM token / KB 上传了多少 字节.

**设计点**:
- **配额表** `workspace_quota`:每个 ws 有 monthly_llm_token_budget / monthly_storage_byte_budget / monthly_meeting_count_budget
- **用量计数**:hook 到 `llm_direct.stream_chat` 完成时,记 token 数 → `workspace_usage` 累计行
- **/super 列表** 加列 "本月 LLM 调用 / token / 大致 ¥"(基于 qwen-max-latest 1元/百万 token 估算)
- **客户自服务** /admin/usage:客户自己看自己的用量 + 剩余配额
- **超限告警**:配额 ≥ 80% 邮件告警,≥ 100% 写端点 429

**风险**:LLM 计费每家供应商不同(qwen-max / deepseek / openai),需要 一个 token→¥ 映射表(`model_provider_config` 加 `price_per_1m_input_token` 字段)

**工时**:~2 天

---

### C4 · 邮件自动发 invite_url(取代手动复制)

**背景**:v26.4 创建客户空间后,你 手动复制 邀请链接 微信发给客户 — 不够自动化.且 不留 邮件 trace.

**设计点**:
- 接 阿里云邮件推送 / 腾讯邮件推送 / SendGrid
- 创建 workspace 时 默认勾选 "发邮件给 owner",填发件 信件模板 + 邀请链接
- 同时 列出"未发送 / 已发送 / 已点击"状态

**工时**:~0.5 天(主要 邮件 SMTP 配置 + 模板)

---

### C5 · expert/member GET read 数据过滤(v26.3.1 P1 缺口 #3)

**背景**:v26.3.1 写端点 严格 leader_or_admin,但 read 端点 (`GET /consensus`, `/agent-messages`)还是 workspace 内 全开.v26.3.1 当时 conscious 决定不做 — 看客户反馈.

**设计点**:
- expert 只看自己 bound AI 涉及的议程的 consensus + agent_messages
- member 不参会 = 完全看不见 ws 的会议数据
- 实现:GET endpoint 加 ABAC 过滤(类似 dashboard.py 已经做的 expert scope)

**触发**:政务客户 提出 "我们 普通员工 看不到 局长会议的内容 才合规"

**工时**:~1 天

---

### C7 · init_db ALTER 加列 没建 FK + ondelete CASCADE 的老坑(技术债)

**背景**:`backend/app/init_db.py` 的 `_COLUMN_MIGRATIONS` 列表对老表加 `workspace_id` 列时,只指定 列类型 (`UUID`),**没建 FK 约束**.SQLAlchemy model 的 `ForeignKey("workspace.id", ondelete="CASCADE")` 是 `create_all()` 第一次建表时才生效;ALTER 加列时 ondelete 没在 DB 里落地.

**现场证据**:v26.4 清空 时 删 40 个非默认 workspace,但 meeting/long_term_memory/agent 等表里 引用这些 workspace 的行 不会 CASCADE 删 → 留下 84 个孤儿(13 meeting + 7 ltm + 64 agent),手工 `DELETE WHERE workspace_id NOT IN (SELECT id FROM workspace)` 才清掉.

**影响**:
- 删 workspace 不会自动 删 子表 → 应用层 大量 手工 cleanup 代码 + 容易漏
- 部分 endpoint(`scorched-earth-reset`、`workspace.delete`)假设 CASCADE 工作,但 实际可能 留孤儿
- 客户数据安全:理论上 客户 ws 删了 应该 干净;但 实际可能 残留 行(违反 GDPR 类合规要求)

**修法**:
1. 跑一次性 migration 脚本:扫所有 含 `workspace_id` 列 的表,补上 FK + `ON DELETE CASCADE`
   ```sql
   ALTER TABLE meeting
     ADD CONSTRAINT meeting_workspace_fk
     FOREIGN KEY (workspace_id) REFERENCES workspace(id) ON DELETE CASCADE;
   ```
2. 同样审计 `user_id` / `agent_id` / `task_id` 等 其它老 FK 列
3. 加 audit 脚本 `backend/scripts/check_fk_health.py`,跑一次 比对 information_schema vs model 定义

**工时**:~1 天(主要 是 调研 + 测试 migration 不破现有 数据)

**优先级**:⭐⭐ (不阻断业务,但 上线前 应该 闭上;否则 客户 workspace 删了 还 留数据 是合规风险)

### C6 · bound_agent_id ↔ primary_user_id 双向一致性

**背景**:v21 加 `WorkspaceMembership.bound_agent_id`("我管哪个 AI"),v26.0 加 `Agent.primary_user_id`("这个 AI 的科室代表是谁").两个字段 独立 维护,实际 应当 互为反向.目前手工 SQL 绑的 4 个 agent / 4 个 user 是 一致的,其他 26 个 agent 没绑.

**设计点**:
- 后台 /admin/team 显示"该 agent 的 primary_user 是 X" + "该 user 的 bound_agent 是 Y";互相点跳;改一边 联动 写另一边
- DB constraint:CHECK(`agent.primary_user_id IS NULL OR (membership.bound_agent_id = agent.id WHERE membership.user_id = agent.primary_user_id)`)
- 一次性 migration 脚本:扫现有数据,把 缺一边 的 自动补齐(选 哪个 agent 是 该 user 的主 bound — 用最早的 created_at)

**工时**:~0.5 天

## 四、不在 v26.5 范围(留 v26.6+ 或永久不做)
