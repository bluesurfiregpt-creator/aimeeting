# v26.3 召集人模式 · 产品 + 技术 设计规范

> **北极星**:召集人只选议程 + 邀请 AI 专家(可不邀真人),系统自动推进、
> AI 互相讨论、产出共识 + 派发 + 沉淀。
>
> **状态**:design,未动手。本文档锁定细节后开 v26.3-01 原型。

---

## 一、产品定位

### 1.1 它解决什么

**传统会议**:
```
真人 5-15 人 + 议程 → 议题 → 真人讨论 → 真人记纪要 → 真人定 task → 真人执行
                                  ↑
                              真人是稀缺资源,经验在脑子里
```

**v26.3 召集人模式**:
```
召集人 1 人 + 议程 + AI 专家 N 个 → 自动讨论 → 自动纪要 → 自动派发 → AI 知识库累积
                                       ↑
                                AI 专家是无限的,经验在知识库里
```

### 1.2 不解决什么

- **不是替代真人会议**(真人会议保留 mode=hybrid/human)
- **不是 解决"AI 比真人聪明"问题**(短期内 AI 的判断在政务领域 < 资深处长)
- **不是降低召集人责任**(决策、签字、对外承担责任 仍然是真人)
- **不是 把所有会议都变成全 AI**(只用于 议程清晰、信息密度高、不必情绪沟通 的场景)

### 1.3 典型使用场景

| 场景 | 模式建议 | 理由 |
|---|---|---|
| 月度政策研究会 | **auto** | 议程明确,各 AI 专家带各自 KB 来分析,召集人选结论 |
| 跨部门协调会 | hybrid (real + AI) | 需要真人代表科室表态,AI 当辅助 |
| 紧急应急会 | human | 现场感、责任要明确,AI 不参与 |
| 例行汇报 | **auto** | 议程死,AI 各汇报本周本部门进展(KB 自动取) |
| 战略策划会 | hybrid | 创造性思维真人主导,AI 提供数据 |

## 二、实体角色与权责

### 2.1 角色清单

| 角色 | 责任 | v26.3 行为 |
|---|---|---|
| **召集人 (Convener)** | 选议程、邀 AI、审决议、对外签字 | 主动方,但介入轻(只看分歧 + 关键节点) |
| **AI 专家 (Agent)** | 用 KB + 记忆 发言、被分配 task | 主体,自主轮发言、引用资料、提出 / 反驳观点 |
| **议程项 (AgendaItem)** | 讨论的最小单元 | 由召集人提供 / 系统模板生成 |
| **议程主持 (Moderator AI)** | 维持秩序、推进议程、收敛共识 | 内置 AI(`Agent.role='moderator'`),非领域专家 |
| **科室账号 (primary_user)** | 主责 AI 接到 task 后实际执行 | 收到 task 通知,事后办理 |

### 2.2 关键不变量

1. **召集人对会议产出负最终责任**。AI 不签字,AI 不背锅,AI 是建议性的。
2. **moderator AI 不接 task**(v26.0 已实现,routing 排除)。
3. **每场 auto meeting 必须有召集人**(召集人 ≡ created_by user)。
4. **AI 专家发言 不污染真人发言记录**(meeting_agent_message vs meeting_transcript 分两个表,v25 已实现)。

## 四、核心算法

### 4.1 整体调度循环(Orchestrator 主循环)

```
auto_meeting_loop(meeting_id):
    auto_state.phase = 'running'
    for agenda_idx, item in agenda:
        update_state(current_agenda_idx=agenda_idx)
        emit_ws_event('agenda_started', {item})
        
        # 4.2 议程项内的发言轮回
        await run_agenda_item(meeting_id, agenda_idx, item)
        
        # 4.3 收敛共识
        consensus = await collect_consensus(meeting_id, agenda_idx)
        emit_ws_event('consensus_collected', consensus)
        
        # 4.4 分歧检查 — 有则等召集人
        if consensus.needs_human_review:
            update_state(phase='consensus_wait')
            emit_notification(召集人, 'auto_meeting_dissent', ...)
            # 等召集人调 /resume-from-consensus 或 /skip
            await wait_for_review(meeting_id, agenda_idx, timeout=...)
            update_state(phase='running')
    
    # 4.5 全议程跑完
    await finalize_meeting(meeting_id)
    # → generate_summary → action_extractor → 自动派给 AI → 沉淀(v26.2)
```

### 4.2 议程项内 发言轮回(turn-taking)

**v26.3 决策(待你确认)**:**议程主持 AI 主导制**

```
run_agenda_item(meeting_id, agenda_idx, item):
    # 0. 邀请的 AI 中选相关者(基于 keywords/KB 相关度 与 item.title 比较)
    relevant_agents = pick_relevant_agents(item, all_invited_agents, top_k=4)
    
    # 1. 议程主持 AI 开场(简述议题)
    moderator_speech = await moderator.intro(item)
    save_message(moderator, moderator_speech, agenda_idx)
    
    # 2. 轮发言循环
    speaker_seq = []  # 已发过言的 agent ids,顺序
    max_turns = 6  # 每议程项最多 6 轮(可配)
    consecutive_no_new_info = 0
    
    while len(speaker_seq) < max_turns:
        # 2a. 选下一个发言 AI
        next_agent = await pick_next_speaker(
            relevant_agents, speaker_seq, recent_messages
        )
        if not next_agent:
            break  # 都说完了
        
        # 2b. 该 AI 看 议程 + 前面所有发言 + 自己 KB,做 streamed reply
        prev_messages = get_messages_for_agenda(meeting_id, agenda_idx)
        agent_reply = await stream_agent_reply(
            next_agent,
            item.title,
            prev_messages,
            instruction="基于自己 KB,补充 / 反驳 / 整合前面发言。简洁,100-300 字。"
        )
        save_message(next_agent, agent_reply, agenda_idx,
                     reply_to=last_message_id)
        speaker_seq.append(next_agent.id)
        
        # 2c. 议程主持 判定是否充分讨论
        if len(speaker_seq) >= 3:  # 至少 3 轮才检查
            adequate = await moderator.judge_adequacy(
                item, get_messages_for_agenda(meeting_id, agenda_idx)
            )
            if adequate.is_adequate:
                # 主持人收尾
                wrap = await moderator.wrap_up(item, messages)
                save_message(moderator, wrap, agenda_idx)
                break
    
    return speaker_seq
```

**关键点**:
- moderator AI 是议程主持人,**它来判定何时收尾**(LLM 判定 + 硬上限 max_turns=6)
- 每个 AI 发言时看到**前面全部消息**,prompt 强制要求"对前面观点 补充 / 反驳"
- max_turns 硬上限避免 AI 互掐没完(政务场景里这种情况罕见但要兜底)

### 4.3 共识收敛

```
collect_consensus(meeting_id, agenda_idx):
    messages = get_messages_for_agenda(meeting_id, agenda_idx)
    
    # 调 qwen-max 双任务:
    #  - 总结 "这个议程项达成了什么共识" (markdown)
    #  - 识别分歧点 (json 列表,每个含 agent_ids + summary)
    output = await llm_collect({
        "agenda": item.title,
        "messages": [...messages],
    })
    
    consensus = MeetingConsensus(
        meeting_id, agenda_idx, item.title,
        consensus_md=output.consensus,
        dissents=output.dissents,
        needs_human_review=len(output.dissents) > 0
    )
    save(consensus)
    return consensus
```

| 暂停时 | 恢复 | `POST /orchestrate/resume` |

### 4.5 prompts 草稿(待 v26.3-01 原型期 调试)

**moderator.intro**:
```
你是会议主持人。当前议程项:"{title}"。
请用 50-80 字简明开场,提出 1-2 个引导问题让在座 AI 专家围绕讨论。
不要给出结论,你只是主持。
```

**agent_reply**:
```
你是 {agent.name},专精 {agent.domain}。
当前议程项:"{title}"。
前面已发言(按时间顺序):
{prev_messages_formatted}

请基于你的知识库 + 经验,做以下其一:
  1) 补充:前面没提到的角度
  2) 反驳:对某位专家观点提出 不同看法 + 理由
  3) 整合:把前面 2-3 个观点 拢成一个执行方案

100-300 字。引用 KB 时用 [资料 X] 角标。
不要客套,直接说观点。不要总结全局,那是主持人的事。
```

**moderator.judge_adequacy**:
```
当前议程:{title}。
已有 {N} 轮发言:
{messages}

请判定这议程项是否已讨论充分(达成共识 或 分歧已清晰):
{
  "is_adequate": true/false,
  "reason": "为啥(简短)",
  "missing_perspectives": ["缺哪类视角",...]  # 若 false
}
```

**collect_consensus**:
```
议程项:{title}。
完整发言:
{messages}

输出 JSON:
{
  "consensus": "<markdown,200-500 字 该议程项达成的共识>",
  "dissents": [
    { "agent_ids": [...], "point": "分歧点", "summary": "..." },
    ...
  ]
}
若无分歧,dissents=[]。
若无共识可达,consensus="(本议程项未达成共识,议程主持收尾时已说明原因)"
```

## 七、关键技术挑战 + 缓解

| # | 风险 | 缓解 |
|---|---|---|
| 1 | LLM 调用密集,成本爆炸 | 每议程项 max_turns=6;moderator 判 adequate 提前收;每 reply 300 字限 |
| 2 | AI 互掐 / 死循环 | moderator 是唯一判 adequate 的;硬上限 6 轮;dissent 触发 wait 召集人 |
| 3 | AI 之间发言重复 | 每 reply 必须看前面消息,prompt 强制 "补充/反驳/整合" 三选一 |
| 4 | 调度器在 worker 进程崩 → 会议挂 | 写 auto_state 持久化;启动 worker 时扫 phase=running 自动 resume |
| 5 | WebSocket 连接断 | 复用 已有 STT/agent ws 基础设施;客户端断重连拉最新 state |
| 6 | LLM 调用超时(2 min+) | 设 60s timeout,超时 emit `auto.agent_timeout`,跳过该 agent 这一轮 |
| 7 | 议程过短 (1-2 项) | 仍然 OK,跑得很快(2-3 分钟一场)|
| 8 | 议程过长 (10+ 项) | 加 整场硬上限 (默认 45 min),超时 紧急 finalize 并通知召集人 |

## 八、性能预估

**单议程项**:
- 主持人 intro: 2-3s
- 6 轮 agent reply × ~10s 每轮 = 60s
- adequacy check 每轮 × 3s = 18s
- consensus collect: 5-10s
- 共 ~90-100s ≈ **1.5 分钟**

**典型 4 议程项 会议**:6-8 分钟

**LLM 成本**(qwen-max,每千 token 0.04 元):
- 单场 ~30 次 LLM 调用 × ~2000 token 平均 = 60K token
- 成本 ~2.4 元/场

## 九、与 v26.0/.1/.2 的协同

```
v26.3 召集人模式
  ├─ 用 v26.0 的 agent-centric 派发(议程末 抽出来的 task 派给主责 AI)
  ├─ 用 v26.1 的 KB embedding 评分(选 relevant_agents 时,KB 相关度)
  ├─ 用 v26.2 的任务办结沉淀(task 跑完后 资料回 AI KB)
  └─ 增加新闭环:    AI 知识库 → 自动参会发言 → 产任务 → 知识库 (循环)
```

**关键洞察**:**v26.3 是 v26.0/.1/.2 的"消费方"**。前面铺的全部基础设施在 v26.3
才真正展现出复利价值 — 没 v26.2 沉淀,AI 永远是"初学者";没 v26.1 KB 检索,
AI 发言空泛;没 v26.0 派发,会议产出无主。
