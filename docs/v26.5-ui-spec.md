## 一、设计原则

1. **UI 上看不到 = 用户不会试**:能权限拦的按钮 / 入口 / tab 在用户看到之前 就**隐藏**,不要 让 用户点了才得到 toast 拒绝
2. **看得到 ≠ 能改**:列表 / 内容 显示给所有人,只 写 按钮 按角色 渲染
3. **视觉解释边界**:不属于自己管的 AI / KB / 记忆,**用图标 + tooltip 说清楚**(`🔒 由 <user> 管理,你无权编辑`),不让用户 自己 猜
4. **降级路径透明**:非超管访问 /super 时跳 / 不一声不响,而是 toast 解释 + 跳转

### 3.2 `/admin/layout.tsx` 守卫

**当前代码**(line 9):
```typescript
const ADMIN_ROLES = new Set(["owner", "admin", "leader"]);
// non-admin → 跳 /me
```

**改成**:
```typescript
const ADMIN_ROLES = new Set(["owner", "admin", "leader", "manager"]);
const FULL_ADMIN_ROLES = new Set(["owner", "admin", "leader"]);  // 用于 tab 渲染
// non-admin (包括 member) → 跳 /me

// Tab 渲染按 role:
function visibleTabs(role: string) {
  if (FULL_ADMIN_ROLES.has(role)) {
    return ["agents", "knowledge", "asr", "models", "memory", "cron", "team", "access", "audit", "demo"];
  }
  if (role === "manager") {
    // 决策点 #1: 我推:agent + knowledge + memory
    return ["agents", "knowledge", "memory"];
    // 决策点 #2: 是否加 "audit" 让 manager 看 自己 agent 相关的日志?
  }
  return [];
}
```

## 四、视觉提示 设计

### 4.1 🔒 锁图标 + tooltip

**用在哪**:
- /admin/agents 卡片右上角(非 primary_user 看到 `🔒`)
- /admin/knowledge 卡片 KB 标题旁(`🔒` + tooltip "此 KB 主要服务 <agent> 由 <user> 管理")
- /admin/memory 同
- 创建会议页 AI 卡片底部小字(`管理人:<user>`)

### 4.2 隐藏 vs 禁用 vs 提示

| 元素类型 | 处理策略 |
|---|---|
| 顶栏入口(⚙️ ⚡ 📊)| **隐藏**(不在 DOM)|
| Tab 入口(/admin 下子菜单) | **隐藏** |
| 列表 + 卡片 | **始终显示**(只读)|
| 修改按钮(编辑 / 删除 / 上传)| **不渲染** + 同位置 显示 `🔒 由 <user> 管理` |
| 表单字段(eg primary_user_id 在 edit modal)| **禁用 + 灰显** + tooltip 解释 |
| 整页跳转(member 进 /admin)| **toast 解释 + 1.5s 跳 /me** |

### 4.3 toast 分类(沿用 v26.4-fix4)

| 后端错误 prefix | toast 颜色 | 标题 | 何时见 |
|---|---|---|---|
| `[权限不足]` | 🟡 黄 warn | "权限不足" | manager 试改别人 primary 的 agent / member 进 /admin |
| `[操作受限]` | 🔵 蓝 info | "操作受限" | 删 workspace owner / 焦土清除 |
| `[资源保护]` | 🟡 黄 warn | "资源保护" | 任务 资源所有权 |
| `[需重新登录]` | 🔴 红 error | "需重新登录" | 账号 disabled / 没 workspace |

## 五、4 个决策点 + 我的推荐

### 决策 #1 · manager 进 /admin 看到 哪几个 tab?

**A**:仅 `agents` + `knowledge` + `memory`(我推荐)
**B**:加 `audit` 让 manager 看自己 agent 相关日志
**C**:加 `team` 让 manager 看 团队全员(但不能改)
**D**:加 `dashboard` 入口 把看板放到 admin 后台里(改造大)

**我推 A**:最小可用,manager 进 admin 只为维护 自己管的 AI / KB / 记忆,其他 tab 是 admin 职责。

### 决策 #2 · /admin/audit 给 manager 看 全 ws 日志 还是 只看自己 agent 相关?

**A**:**完全不给 manager 看**(audit 是 admin 职责)
**B**:看 全 ws 日志,但 filter 默认 自己 agent 相关
**C**:看 limited 视图(只 own agent 的 kb.upload / agent.update / task 相关 log)

**我推 A**:audit 是 合规视角,manager 不该看 其他部门的 操作日志。决策 #1 已没 audit tab,跟这里一致。

### 决策 #3 · member 能进 /admin 吗?

**A**:**完全不能**(我推)— 顶栏 ⚙️ 隐藏,直接 URL 输 /admin/* 跳 /me
**B**:能进 但 全部 tab 灰禁

**我推 A**:零暴露面,member 不该有 任何 admin 入口。

### 决策 #4 · /meeting/[id]/orchestrate 召集人控制台 manager 的权限?

**A**:**只读**(跟 expert / member 一样,v26.4-fix1 当前行为)
**B**:能控制(启动 / 暂停 / 取消),理由"是部门 AI 管理人"

**我推 A**:auto 会议是跨部门决策,只 leader+ 控制。manager 看自己 AI 在会上发言就够了。

### 决策 #5 · manager 创建 agent / KB / memory 行不行?

**A**:**不行**(P0 严控,leader+ 才能创建)
**B**:能创建,自动设 primary_user_id = caller(自己管自己创建的)

**我推 A**:agent / KB 是 workspace 资产,创建权限 收口在 leader+ 一处,免得 N 个 manager 各自乱创造成 KB 碎片化。
