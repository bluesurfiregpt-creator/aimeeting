### Root cause: prod nginx 漏配 `location /ws/`

```
2026/05/27 15:15:56 [error] upstream timed out (110) while reading response
header from upstream, client: ..., request: "GET /ws/stt?meeting_id=...",
```

repo 里的 `deploy/nginx/aimeeting.conf` 一直有 `location /ws/`,但 prod active

## 4. 经验沉淀 (给下次的人)

### 4.1 测之前 **必须** 校时

**不仅校 git commit,还要校 prod container 重建时间**。Kimi v1 的 8 分钟测试白跑,
是因为没人确认 deploy script 跑完的具体时间 跟 测试开始时间 的先后。

v2 Kimi 用例 P-6 解法:用 frontend `<div data-testid="version-badge"
data-build-version="...">` 时戳(Next.js build 时 inject)做硬验。这是 工程师 / Kimi
都能直接 curl 看到的事实。

**给所有 deploy 后 Kimi 测试模板加一条 "P-时戳" 预检** — 任何 Kimi 报告里 测试时间 必须 >
deploy 时戳 + 安全 buffer。

### 4.2 nginx 漏配的隐蔽性

WS 漏 location 后,行为是:
- 浏览器看到的是 "WS 反复重连 + UI 数据正常" → 因为 polling fallback
- backend log 看不到 WS 连接(连接被 nginx 截到 frontend 去了)
- nginx error log 有 "upstream timed out" 但写的是 frontend `:3000` 路径

**症状跟应用层 bug 高度相似**(都是 "WS 连不上 + 数据靠 polling")。区分方法:
1. **看 backend log 有没有 `WebSocket /ws/...` accept 记录** — 没有的话,问题不在
   backend,在 nginx 之前
2. **看 nginx access log 看 `/ws/stt` 的 status** — 499 / 504 → nginx 层面已经
   超时,根本没透到 backend
3. **直接 `curl -i -H "Connection: Upgrade" ... wss://...` 看响应**:
   - `101` → nginx → backend 整链通
   - `200` / timeout → nginx 层路由错
   - `403` → 路径不对 / backend route 不存在

下次 deploy 完应该跑一次 smoke "curl WS upgrade",作为 deploy script 的 post-check。
**TODO (Sprint 4+ 可选)**:在 `deploy/rsync-up.sh` 末尾加一个 curl 101 verification。

### 4.3 nginx config 不入 deploy 流程的坑

repo 里的 `deploy/nginx/aimeeting.conf` 跟 prod active config (`/etc/nginx/
sites-available/aimeeting`) 长期分裂。原因:

- `deploy/bootstrap.sh` 是 **first-time setup** 才跑的脚本,日常 `rsync-up.sh
  --deploy` 不会重写 nginx config
- 当时 bootstrap.sh 写到 `aimeeting.conf` (带 .conf 后缀),但 prod active 是
  `aimeeting` (无 .conf,可能 手动 setup 时漏了)
- 之后任何人在 prod 手动改了 active config,都不会反向同步回 repo
- 任何人改了 repo,也不会 sync 到 prod (除非重跑 bootstrap)

**本次修复**:
1. 修了 bootstrap.sh line 73-78,目标路径改成 `aimeeting`(无 .conf)对齐 prod
   active 文件名 — 下次 server 重装跑 bootstrap 能落对位置
2. nginx config 同步 仍是 first-time setup 行为,**没**加日常 deploy sync — 因
   为 prod nginx config 可能有 server-only 调优(`client_max_body_size 200M` /
   `proxy_read_timeout 600s` 等),粗暴覆盖 风险高于 收益

**长期建议** (不在 Sprint 3 范围,留 backlog):
- 把 nginx config 做成 配置注入 (env / template),deploy 时按 prod 实际跑的
  port + cert 路径 渲染,然后 sync。但要 PM 拍板 + 增量做。
