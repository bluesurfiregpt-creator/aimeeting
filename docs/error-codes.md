# Aimeeting · 错误提示 分类参考

> 自 v26.4-fix4 起,后端所有 4xx 错误的 `detail` 字段 用 `[类别] 描述` 格式标分类.
> 前端 toast 根据 类别 自动选 颜色 + 措辞,让用户能区分 "我没权限" / "系统设计如此" / "Bug".

---

## 4 个类别

| 类别 prefix | 含义 | toast 颜色 | 标题 | 用户应理解为 |
|---|---|---|---|---|
| `[权限不足]` | RBAC 角色不够 (member 调 leader 端点 等) | 🟡 warn | "权限不足" | 这是 ABAC 设计 — 你 当前角色 不能做这事.找管理员升级角色 或 换有权限的账号. |
| `[操作受限]` | 业务规则故意拒绝 (mode/phase/状态 不匹配,或安全保护设计) | 🔵 info | "操作受限" | 系统设计如此,**不是 bug**.例:已结束会议 不能再启动 / 焦土清除 仅 owner / 不能直接删 workspace owner. |
| `[资源保护]` | 资源所有权检查 (不是 assignee / author / 申请审批人) | 🟡 warn | "资源保护" | 这资源 不归 你 直接管.如果你 觉得应该归你,联系任务发起人或管理员 改派. |
| `[需重新登录]` | 账号状态异常 | 🔴 error | "需重新登录" | 账号 disabled / 没 workspace / 数据异常.退出重登,如仍报错联系管理员. |

> 没 prefix 的 4xx 是 老错误 (或第三方错误透传),会显示 `请求失败 (4xx)`.遇到 长期未消失的"无 prefix 4xx"请告诉开发组排查.

## 给客户演示时怎么解释

如果客户演示过程中看到:

- 🟡 **"权限不足"** → "您 当前角色 不能做这事,这是 ABAC 安全设计.我换管理员账号给您看".
- 🔵 **"操作受限"** → "**这不是 bug**.系统故意保护 — 比如焦土清除只有 workspace owner 能做,平台运营都不能代您操作".
- 🟡 **"资源保护"** → "这条任务不归您,只有任务负责人能操作".
- 🔴 **"需重新登录"** → "账号状态异常,需要退出重登".

**关键话术**:`[操作受限]` 是产品 **加分项** — 客户会觉得"哦,这系统真的有边界感,不会乱给权限".

## 关于 platform admin 的特殊行为

v26.4 平台超管 (env `PLATFORM_ADMIN_EMAILS`) 切换到客户 workspace 后:

- ✅ 进入大多数 admin 端点 不报 `[权限不足]`(`is_leader_or_admin` 已 short-circuit,v26.4-fix1/fix3)
- ⚠️ 仍会撞:
  - `[操作受限]` 焦土清除 (设计安全 — 即使是超管也不代客户做)
  - `[资源保护]` 任务负责人检查 (超管不是任务 assignee)
  - `[操作受限]` 改/删 客户 workspace owner (即使是超管也不能误操作)

这些**故意保留** — 防止运营误操作客户数据.

## 给开发者:加新 403 的规范

新加任何 `raise HTTPException(403, ...)` 必须用 `[类别] 描述` 格式.4 个类别按上表选:
- 如果是 "你 当前角色不够" → `[权限不足]`
- 如果是 "业务规则不让 / 安全保护" → `[操作受限]`
- 如果是 "这资源不归你" → `[资源保护]`
- 如果是 "账号本身有问题" → `[需重新登录]`

加完 detail 后,**更新本文档** 对应表格.
