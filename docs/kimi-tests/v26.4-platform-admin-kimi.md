# Kimi 测试用例 · v26.4 Platform Admin

> 🚨 **首要前提**:服务器 `/opt/aimeeting/backend/.env` 必须 已加
> `PLATFORM_ADMIN_EMAILS=bluesurfiregpt@gmail.com` 才能进 /super.
> 如果 env 没配,**bluesurfiregpt** 本人 进 /super 也会 403.
>
> 部署方步骤:
> 1. `nano /opt/aimeeting/backend/.env` 末尾加 `PLATFORM_ADMIN_EMAILS=<email>`
> 2. `cd /opt/aimeeting/deploy && docker compose up -d --force-recreate backend`
>    (⚠️ 必须 force-recreate.`docker compose restart` 不重读 env_file,改了不生效 — 踩过坑)
> 3. 验证:`docker exec aimeeting-backend env | grep PLATFORM` 应输出 `PLATFORM_ADMIN_EMAILS=...`
>
> **变更**:`feat(v26.4)` 跨 workspace 平台超管 SaaS 平台层 — 列表 / 创建 / 切换 三件套.
> **commit**:本批 (见 git log)

> **测试对象**:Kimi
> **目标**:验证 ⚡ 平台超管的 list / create / switch 端到端 + 非超管 403 拒绝.

## 2. 预检 P-1 ~ P-3(任一 fail → 后续 BLOCKED)

### P-1 服务可达
`curl -sS -o /dev/null -w "%{http_code}\n" https://aimeeting.zhzjpt.cn/` 必字面 `200`.

### P-2 后端活
`curl -sS -o /dev/null -w "%{http_code}\n" https://aimeeting.zhzjpt.cn/api/agents` 必字面 `401`.

### P-3 平台超管 登录 + env 配置 验证
1. 浏览器 1 用 `bluesurfiregpt@gmail.com` 登录
2. 调:
   ```bash
   curl -sS https://aimeeting.zhzjpt.cn/api/super/me -b "<COOKIE>"
   ```
3. response 必须 `{"is_platform_admin": true, "email": "bluesurfiregpt@gmail.com", "platform_admin_emails_count": N}` (N ≥ 1)

**Fail**:
- `is_platform_admin: false` → env 没配,**整份 BLOCKED**,联系部署方在 `.env` 加 `PLATFORM_ADMIN_EMAILS=bluesurfiregpt@gmail.com` 并 `docker compose restart backend`
- `platform_admin_emails_count: 0` → env 解析失败

**必须复述**:完整 JSON

**必须复述**:API 返回的 第一个 element 的完整 JSON

### T-03 · 非超管 调 /api/super/workspaces → 403

**前置**:P 全 pass,你 有非超管 owner cookie
**步骤**:
1. 浏览器 2 用 非超管 owner 登录
2. F12 复制 cookie
3. curl:
   ```bash
   curl -sS https://aimeeting.zhzjpt.cn/api/super/workspaces -b "aimeeting_session=<NON_SUPER_COOKIE>" -w "\n[status=%{http_code}]\n"
   ```

**Pass 条件**:
- status 字面 `403`
- body 字面 含 `平台超管` 或 `PLATFORM_ADMIN_EMAILS`

**Fail**:

- 401 → cookie 错,重取

**必须复述**:完整 response

### T-05 · 创建 workspace + 一次性邀请链接

**前置**:T-02 pass,在 platform admin 视角 /super 页
**步骤**:
1. 点 ➕ 新建工作空间(testid `open-create`)
2. modal 弹出.填:
   - 工作空间名称:`kimi-v26.4-T05-<时间戳>` (例 `kimi-v26.4-T05-20260513200000`)
   - owner 姓名:`Kimi Test Owner`
   - owner 邮箱:`kimi-v26.4-T05-<时间戳>@example.com`(不要 重复 用 现有邮箱)
   - 临时密码:**留空**(让系统自动生成 32 位)
   - "生成一次性邀请链接":勾上
   - "自动 seed 演示数据":不勾(避免 ~60s 等待)
3. 点 ✓ 创建
4. 等 < 3 秒
5. 看回执屏

**Pass 条件**(全部):
- 出现 emerald 回执屏,标题 字面含 `✓ workspace 「kimi-v26.4-T05-...」已创建`
- 显示 字段:
  - owner 邮箱(字面 = 你填的)
  - 临时密码(字面 ≥ 24 字符,是随机串)
  - 一次性邀请链接(字面以 `https://aimeeting.zhzjpt.cn/register?invite=` 开头)
- 点 "复制" 按钮 toast 显示 `已复制邀请链接`
- 点 关闭 后 列表刷新,出现 新的 workspace 一行

**Fail**:
- 报错 → 粘 error 原文
- 临时密码 或 邀请链接 缺失 → fail

**必须复述**:回执屏 完整 截图 + 三个凭证 的完整文字(记下来后续 T-07 用)

### T-06 · 切换 workspace · session 真切换 + audit

**前置**:T-05 pass,已有新 workspace
**步骤**:
1. 在 /super 列表 找到 你 主 workspace 之外的某行(如 T-05 创建的那个)
2. 点 "进入 →"(testid `switch-<slug>`)
3. confirm 弹窗 点确定
4. 等 ~1 秒,页面 跳到 `/`
5. F12 看 cookie:`aimeeting_session` value 应已 改变
6. 调:`curl -sS https://aimeeting.zhzjpt.cn/api/auth/me -b "<NEW_COOKIE>"` 看返回的 `workspace_id` 是否 是 目标 workspace 的 id
7. (audit 验证)切回原 workspace 看 audit 是否记到 目标 workspace

**Pass 条件**:
- 跳转 / 后 顶栏的 workspace_name 显示 字面 = T-05 那个新创建 workspace 名字
- `/api/auth/me` 返回的 `workspace_id` 字面 = T-05 创建出来的 workspace_id
- 列表里 `meeting` / `agent` 数 现在 = 0 (新 workspace 刚建,啥都没有)

**Fail**:
- workspace_name 还是原来的 → fail("issue_token 没切 wsid")
- workspace_id 不对 → fail("set_session_cookie 没生效")

**必须复述**:`GET /api/auth/me` 切换前 / 切换后 的两份 workspace_id
