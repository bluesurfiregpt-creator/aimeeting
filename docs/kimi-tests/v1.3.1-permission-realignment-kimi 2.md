## 3. 预检 P-1 ~ P-4

### P-1 服务器健康

- **步骤**: `curl -s -o /dev/null -w "%{http_code}" https://aimeeting.zhzjpt.cn/`
- **Pass**: 返回 `200`.
- **Fail**: 非 200 → 后续全 BLOCKED.

### P-2 后端 API 活

- **步骤**: `curl -s -o /dev/null -w "%{http_code}" https://aimeeting.zhzjpt.cn/api/auth/me`
- **Pass**: 返回 `401` (没 cookie). 不允许 5xx.
- **Fail**: 5xx 或 connection refused → BLOCKED.

### P-3 登录 system_owner

- **步骤**:
  1. `POST /api/auth/login` with `{"account": "bluesurfiregpt@gmail.com", "password": "aimeeting123"}`
  2. `GET /api/auth/me` 带 cookie
- **Pass**: P-3.1 返回 200 + body 含 `"role"`. P-3.2 返回 200 + body 含 `"email": "bluesurfiregpt@gmail.com"`. role 字面值 = `workspace_creator` (新值, 不该是 `owner`).
- **Fail**: role 仍是 `owner` (说明 init_db migration 没跑).
- **必须复述**: 完整 me JSON 原文.

### P-4 登录其他 4 个 demo 账号 (逐一)

- 步骤同 P-3, 对 demo.lijg / demo.chensy / demo.fengl / demo.hanx 各跑一次.
- **Pass**:
  - demo.lijg → role = `leader`
  - demo.chensy → role = `admin`
  - **demo.fengl → role = `agent_owner`** (不该是 `manager` 或 `expert`)
  - demo.hanx → role = `member`
- **Fail**: 任何一个 demo 账号 role 还是旧值 (manager / expert) → init_db migration 没跑, BLOCKED.

## 4. 测试用例 T-01 ~ T-10

### T-01: 普通用户注册 → 加入 demo ws 当 member (决策 4)

- **前置**: P-1~P-4 全 pass.
- **目标**: 验证 决策 4 — 普通用户 register 不带 invite 不再 自动建新 ws 当 owner, 改成 加入 demo ws 当 member.
- **步骤**:
  1. 生成一个唯一邮箱 `kimi-test-${ts}@example.com` (ts = unix 时间戳).
  2. `POST /api/auth/register` with `{"email": "kimi-test-${ts}@example.com", "password": "kimitest123", "name": "Kimi 测试"}`.
  3. 收到 MeOut response 后, 读 `role` + `workspace_slug` 字段.
- **Pass 条件**:
  - HTTP 状态 `200`.
  - `role` = `"member"` (字面值).
  - `workspace_slug` = `"default"` (字面值).
- **Fail 条件**:
  - role = `"owner"` 或 `"workspace_creator"` → 决策 4 失败, 普通注册仍然建新 ws.
  - workspace_slug != `"default"` → 新 ws 被错误创建.
- **必须复述**: 完整 response JSON 原文.

### T-02: system_owner 注册 → 建新 ws 当 workspace_creator (决策 4)

- **前置**: T-01.
- **目标**: 验证 决策 4 — system_owner (env 白名单) 注册时仍可建新 ws.
- **步骤**:
  1. 由 system_owner 联系运维确认 `kimi-sys-owner-test@example.com` 在 env `PLATFORM_ADMIN_EMAILS` 白名单. (如未配置, 标 BLOCKED + 说明).
  2. `POST /api/auth/register` with `{"email": "kimi-sys-owner-test@example.com", "password": "kimitest123", "name": "Kimi SO 测试", "workspace_name": "Kimi 测试 SO ws"}`.
- **Pass 条件**:
  - HTTP `200`.
  - `role` = `"workspace_creator"`.
  - `workspace_slug` != `"default"` (建了新 ws).
- **Fail 条件**:
  - role = `"member"` (退到了 demo ws) → 白名单未生效.
  - HTTP 403 → 白名单未生效.
- **观察**: 如果 env 白名单只配了 bluesurfiregpt@gmail.com, 本用例标 SKIP + 说明.

### T-03: leader 改 AI 配置 → 200 (PM Q7.4)

- **前置**: P-3, P-4.
- **目标**: 验证 workspace_manager (leader) 仍可改 AI.
- **步骤**:
  1. 登录 demo.lijg (leader).
  2. `GET /api/agents?active_only=true` 拿任一 agent id.
  3. `PATCH /api/agents/{id}` with `{"persona": "Kimi 测试: leader 改 AI persona, ${ts}"}`.
- **Pass**: HTTP `200`, response.persona 包含 `"Kimi 测试"`.
- **Fail**: HTTP 403 → leader 不该被拦.
- **必须复述**: response 中 `name` / `persona` 字段 JSON 原文.

### T-04: admin 改 AI 配置 → 403 (PM Q7.4 严控)

- **前置**: P-3, P-4.
- **目标**: 验证 admin 不能编辑 AI (本次重点变化).
- **步骤**:
  1. 登录 demo.chensy (admin).
  2. `GET /api/agents?active_only=true` 拿任一 agent id (不该是 demo.chensy primary 的, 因为 demo.chensy 应该没绑 primary).
  3. `PATCH /api/agents/{id}` with `{"persona": "Kimi 测试: admin 改 AI, ${ts}"}`.
- **Pass**: HTTP `403`. 错误消息含 `"workspace_creator"` 或 `"leader"` 或 `"agent_owner"`.
- **Fail**: HTTP 200 → admin 仍可改 AI, v1.3.1 决策 3 失败.
- **必须复述**: HTTP status + detail 字段 JSON 原文.

### T-05: agent_owner 改自己 primary AI → 200 (PM Q7.4 row-level)

- **前置**: P-3, P-4. demo.fengl 是 物业监管AI专家 的 primary_user.
- **目标**: 验证 agent_owner 可改自己 primary 的 AI.
- **步骤**:
  1. 登录 demo.fengl (agent_owner).
  2. `GET /api/agents?q=物业监管` 拿到该 AI id.
  3. `PATCH /api/agents/{id}` with `{"persona": "Kimi 测试: agent_owner 改自己 AI, ${ts}"}`.
- **Pass**: HTTP `200`.
- **Fail**: HTTP 403 → agent_owner 写自己 primary AI 也被拦, ABAC 失败.
- **必须复述**: response JSON 原文.

### T-06: agent_owner 改别人 AI → 403 (PM Q7.4 边界)

- **前置**: T-05.
- **目标**: 验证 agent_owner 不能改别人的 AI.
- **步骤**:
  1. 登录 demo.fengl (仍 agent_owner).
  2. `GET /api/agents?q=房屋安全` 拿到 房屋安全 AI 的 id (不该是 demo.fengl primary).
  3. `PATCH /api/agents/{id}` with `{"persona": "Kimi 测试: 改别人 AI, ${ts}"}`.
- **Pass**: HTTP `403`. detail 含 `agent_owner` 或 `primary_user`.
- **Fail**: HTTP 200 → agent_owner 越权改了别人的 AI.

### T-07: admin 创建 AI → 403 (PM Q7.4)

- **前置**: P-4.
- **目标**: 验证 admin 不能创建 AI (本次重点变化).
- **步骤**:
  1. 登录 demo.chensy (admin).
  2. `POST /api/agents` with `{"name": "Kimi 测试 AI ${ts}", "domain": "测试", "persona": "测试"}`.
- **Pass**: HTTP `403`. detail 含 `workspace_creator` 或 `leader`.
- **Fail**: HTTP 200 → admin 创建了 AI, 决策 3 失败.

### T-08: member 列团队 → 403 (admin_or_above)

- **前置**: P-4.
- **目标**: 验证 member 不能列 team members.
- **步骤**:
  1. 登录 demo.hanx (member).
  2. `GET /api/team/members`.
- **Pass**: HTTP `403`. detail 含 `workspace_creator / leader / admin`.
- **Fail**: HTTP 200 → member 看了全员邮箱, ABAC 失败.

### T-09: admin 邀请新成员 (leader role) → 200 + invite_url (admin_or_above)

- **前置**: P-4.
- **目标**: 验证 admin 仍可邀请成员 (这是 admin 保留的 ws_admin_or_above 权).
- **步骤**:
  1. 登录 demo.chensy (admin).
  2. `POST /api/team/invitations` with `{"email": "kimi-invite-test@example.com", "role": "agent_owner"}`.
- **Pass**: HTTP `200`. response 含 `invite_url` 字段 + `role` = `"agent_owner"`.
- **Fail**:
  - HTTP 403 → admin 不能邀请, 但 v1.3.1 决策 3 admin 应保留邀请权.
  - HTTP 400 `"role must be one of"` → 后端 _INVITABLE_ROLES 没加 agent_owner.

### T-10: member 调 orchestrate 写端点 → 403

- **前置**: P-4.
- **目标**: 验证 member 不能触发 auto 会议 orchestrate (workspace_admin_or_above).
- **步骤**:
  1. 登录 demo.hanx (member).
  2. 先 `POST /api/meetings` with `{"title": "Kimi 测试会议 ${ts}", "mode": "hybrid"}`. (member 可创 hybrid 会议).
  3. 拿到 meeting id 后 `POST /api/meetings/{id}/orchestrate/start` with `{}`.
- **Pass**:
  - 步骤 2 HTTP `200` (member 可发起 hybrid 会议).
  - 步骤 3 HTTP `403`.
- **Fail**: 步骤 3 HTTP 200 → member 调了 orchestrate 写端点, 越权.
