# Kimi 测试用例 · v27.0-mobile P4.2 — 召 AI 发言 + 结束会议

## 顶部 6 条死规矩 (反幻觉)

1. 只看真实页面 / 真实 HTTP 响应.
2. 每步判定必须基于本步实际看到的数据.
3. 复述要原文 (带引号的精确字符串).
4. 不允许跳步. 前一步 fail → 后续 BLOCKED.
5. 不允许编造账号.
6. 每用例 4 段必须给齐: 实际看到 / 判定 / 失败理由 / 证据.

## 环境 与 账号

- 站点根: `https://aimeeting.zhzjpt.cn`
- 涉及页面: `/m/meetings/<meeting_id>`
- 涉及 API:
  - `GET /api/m/meetings/{meeting_id}` (现在多返 `attending_agents`)
  - `POST /api/m/meetings/{meeting_id}/summon` body `{agent_id, query?}`
  - `POST /api/meetings/{meeting_id}/finalize`

| 角色 | 邮箱 | 密码 |
|------|------|------|
| owner | `bluesurfiregpt@gmail.com` | `<SYSTEM_OWNER_PWD>` |

### 数据库前置假设
- 至少 1 个 status=ongoing 的 meeting, 我 (owner) 有 controller 权限.
- 该 meeting 至少邀请过 1 个 AI 专家 (MeetingAttendee.agent_id IS NOT NULL).

> 若没有 ongoing meeting, T-* 全 BLOCKED. 可在桌面端新建一个会议并邀请 1 个 AI 专家以满足前置.

## 预检 P-1 ~ P-3

### P-1 · summon endpoint 未登录返 401
```
curl -i -X POST https://aimeeting.zhzjpt.cn/api/m/meetings/00000000-0000-0000-0000-000000000000/summon -H 'content-type: application/json' -d '{"agent_id":"00000000-0000-0000-0000-000000000000"}'
```
**Pass**: 401. **Fail**: 5xx / 200.

### P-2 · finalize endpoint 未登录返 401
```
curl -i -X POST https://aimeeting.zhzjpt.cn/api/meetings/00000000-0000-0000-0000-000000000000/finalize
```
**Pass**: 401.

### P-3 · 登录 + 进 ongoing 会议详情 + 校验 attending_agents 字段
- 登录 owner.
- 进 /m → segment「会议视角」 → 「进行中」tab → 找一个 ongoing 会议点进去.
- URL = `/m/meetings/<id>`.
- devtools 抓 `GET /api/m/meetings/<id>` 响应, 看 `attending_agents` 字段.
- **Pass**: 字段存在, 是数组. 数组长度 ≥ 1 (至少 1 个邀请的 AI). 数组每条含 `agent_id, name, nickname, domain, color, role` 六个字段.
- **Fail**: 字段缺失 / 类型错 / 长度 = 0 (说明没邀请 AI, 改去满足前置).

## T-01 · 召 AI sheet 打开 + 列表

### 前置 P-* PASS.

### 步骤
1. 在 ongoing 会议详情页底部, 找到 sticky bar.
2. 点 "💬 召 AI" 按钮 (默认 sticky bar 的副按钮).
3. 观察:
   - 屏幕底部弹起 sheet, 上滑 70-80% 屏高.
   - sheet 顶部有把手 + 标题 "请 AI 专家发言" + 副标题 "选一位, 立刻基于刚才的讨论给出意见".
   - 列表显示所有 `attending_agents` 项, 每项一行: 色块圆头 ◆ + nickname/name (大字 16px) + domain (小字 13px).
   - 底部一个 textarea, placeholder "(可选) 给 AI 一句话提示, 不写走默认".
   - 底部两个按钮: 取消 (灰边) + 蓝色主按钮 "选一位 AI" (disabled, 因为还没选).

### Pass (全部)
- sheet 出现.
- 列出的 agent 数量 == `attending_agents.length` (devtools 验).
- 主按钮文案 "选一位 AI", disabled (opacity 降低).

### Fail
- sheet 没出现 (按钮点了没反应).
- 显示了 alert "Phase 4.2..." (说明没上 P4.2).
- 列表 agent 数量跟 JSON 不一致.

### 必须复述
- 整个 sheet 截图.
- `attending_agents` 完整 JSON 粘贴 (含每条所有字段).

## T-02 · 选一个 AI → 主按钮变文案 → 派发

### 前置 T-01 PASS.

### 步骤
1. 在 sheet 里点一个 agent 项 (任意).
2. 观察:
   - 被选项右侧出现 ✓, 边框变蓝色 (accent-500/60), 背景轻微高亮.
   - 底部主按钮文案变成 "派 X 发言" (X = 该 agent 的 display name).
   - 主按钮 enabled (颜色饱满).
3. 点主按钮.
4. 观察:
   - 按钮文案立即变 "派发中…", disabled.
   - 几百毫秒后 sheet 收起.
   - 屏幕底部出现绿色 toast: `✓ 已请 <agent_name> 发言, 几秒后刷新看回复`.
5. 等约 5-10 秒, 主区域 CurrentTopicCard 的 "AI 智囊" 数应该 +1 (该 AI 的回复进了 insight 列表) 或 真人表态区出现该 AI 名字的发言.

### Pass (全部)
- sheet 收起.
- toast 绿色, 文案含 "已请" + agent_name + "发言".
- 几秒后 refetch `GET /api/m/meetings/<id>`, `current_topic_insights` 长度比点击前 +1 (或 AI 写入 MeetingAgentMessage, 可走 `/api/meetings/<id>/agent-messages` 验).

### Fail
- toast 红色 / 显示错误.
- sheet 没收起.
- 后端 5-30 秒后无新内容 (说明 invoke_agent_directly 失败 — 看 backend log).

### 必须复述
- 点击前 + 5 秒后两次 `GET /api/m/meetings/<id>` 的 `current_topic_insights` 数 (前 vs 后).
- toast 截图 + sheet 收起后的页面截图.

## T-03 · 召不在会议里的 AI 应该拒绝

### 前置 T-01 PASS.

### 步骤
- devtools 直接调用:
```
fetch('/api/m/meetings/<meeting_id>/summon', {
  method:'POST', credentials:'include',
  headers:{'content-type':'application/json'},
  body: JSON.stringify({agent_id:'<某个本 ws 内但没邀请到此 meeting 的 agent_id>'})
}).then(r=>r.json()).then(console.log)
```
- 找一个没邀请到此 meeting 的 agent_id: 看 `GET /api/m/agents/workboard` 全部 agents, 排除 `attending_agents` 里的, 取一个差集.

### Pass
- HTTP 400, body `{"detail":"agent not invited to this meeting"}`.

### Fail
- 200 (说明校验缺失, 安全隐患).

### Skip 允许
- workspace 总共只有 1 个 AI, 全部都在会议里 — 标 SKIP.

## T-04 · 议程全完成 → sticky 显示「结束会议」

### 前置
- 一个 ongoing meeting, 议程已被「推进议程」推到最后一项之后 (`is_agenda_complete=true`).
- 若不存在, 现场先在某个 ongoing meeting 反复点「推进议程」直到议程走完.

### 步骤
1. 进该 meeting 详情页.
2. 看 sticky bar 底部.

### Pass
- sticky bar 不再显示 "推进议程", 改显示 "结束会议" 按钮 (单按钮, 红色或蓝色都行).
- 主区域出现绿色提示 "议程已全部完成".

### Fail
- 仍显示推进议程.
- 没显示结束会议按钮.

## T-05 · 点结束会议 → confirm 对话框

### 前置 T-04 PASS.

### 步骤
1. 点 "结束会议".
2. 观察:
   - 屏幕中央出现 modal dialog (居中浮窗 + 黑色遮罩).
   - 标题 "结束这场会议?".
   - 正文 "结束后 AI 会自动生成会议纪要 + 抽待办 + 提候选记忆. 这一步不可撤销."
   - 两个按钮: "再开一会" (灰) + "结束" (红色 danger).
3. 先点 "再开一会".

### Pass
- dialog 出现, 文案完全一致.
- 点 "再开一会" → dialog 消失, 页面回到原状, 后端 status 仍 ongoing.

### Fail
- 没弹 dialog, 直接调 API.
- 文案不一致.
- 点 "再开一会" 触发了 finalize (bug).

## T-06 · 确认结束 → finalize 调用 + 跳回首页

### 前置 T-05 PASS.

### 步骤
1. 再次点 "结束会议", dialog 出现.
2. 这次点红色 "结束".
3. 观察:
   - "结束" 按钮变 "处理中…" disabled.
   - 几百毫秒后:
     - toast 绿色 "会议已结束, AI 正在生成纪要 + 抽待办".
     - 1-2 秒后页面自动跳到 `/m` (首页).
4. 在 /m 首页, 该 meeting 不应再出现在 Hero "你正在推进" 卡里.
5. 直接 `curl GET /api/m/meetings/<id>`, 看 `status` 字段.

### Pass (全部)
- toast 绿色 + 文案匹配.
- URL 自动变成 `/m`.
- 后端 `meeting.status` 从 "ongoing" → "finished".
- 首页 Hero 卡列表 (`ongoing_meetings`) 不再含该 meeting_id.

### Fail
- 后端 status 未变.
- 没跳回首页.
- toast 红色错误.

### 必须复述
- finalize 前 + 后两次 GET `/api/m/meetings/<id>` 的 `status` 字段.
- toast 截图 + 跳回首页后的截图.
