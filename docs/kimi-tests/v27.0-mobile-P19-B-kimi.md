## T-01 · TXT 文件 上传 → 同步 ready

### 前置 P-* PASS.

### 目标
小文件 (< 2MB) 走同步抽取, 上传完 立刻 `extract_status: "ready"`.

### 步骤
1. 在 `/m/meetings/new`, 点 `+ 添加文件` → 选 `fictional-Q1.txt`.
2. devtools Network 看 POST `/api/meetings/attachments` request.
3. 等 < 10 秒 (LLM summary 调用), 看 response.

### Pass
- request payload 是 `multipart/form-data`, 含 `file` + `client_draft_id` 两个 field.
- response status `200`. body 是 JSON, 字段:
  - `id` (uuid)
  - `filename: "fictional-Q1.txt"`
  - `extension: "txt"`
  - `size_bytes` 等于本地文件 字节数
  - `extract_status: "ready"` (字面)
  - `extract_summary` 非空, 含中文摘要, 包含至少 1 个 brief 里关键词
    (`Q1` / `投诉` / `安保` / `卫生` / `1287` / `35%` / `50w` 等)
- UI: 文件出现在 attachment 列表里, 右侧 chip `✓ 就绪` (emerald 色, 字面命中).

### Fail
- response 503 → OSS 没配, 让用户 配后重试.
- `extract_status: "extracting"` (小文件 不应异步, 说明 SYNC_EXTRACT_THRESHOLD 配错).
- `extract_summary` 全是 brief 里没出现 的词 (说明 LLM call 没拿到 text).

### 必须复述
- POST request 完整 headers + form fields (无 file 二进制).
- response body 完整 JSON 原文.
- attachment 列表项 截图.

## T-05 · GET draft 下列出 多个 attachments

### 前置 T-01 + T-04 都成功 (同一 draft, 已 上传 ≥ 2 份).

### 步骤
- devtools Console 直接 fetch:
  ```js
  // 从 T-01 / T-04 的 response 里 拿 client_draft_id, 这里 用变量 D 占位
  const D = "<同一 draft uuid>";  // T-01 / T-04 的 form field 同一个值
  fetch(`/api/meetings/attachments?draft_id=${D}`, { credentials: "include" })
    .then(r => r.json()).then(console.log);
  ```

### Pass
- response status `200`. body `{ items: [...] }`.
- items 数组长度 ≥ 2.
- items 顺序 按 created_at 升序 (T-01 在 T-04 前).
- 每项 含 filename / extract_status / extract_summary 字段.

### Fail
- items 数 = 0 → draft_id 跟上传时不一致 (前端 draftId 持久性 BUG).
- items 多了别人的文件 → 隔离逻辑漏了 (workspace_id / uploader_user_id 没过滤).

### 必须复述
- response body JSON 完整 原文.

## T-06 · AI 拆议程 用 attachments

### 前置 T-01 / T-04 PASS.

### 目标
attachment 已上传 → AI 拆议程 出来的 议程 + note 真引用 文件内容关键词
(数据 `1287 件 / 安保 / 卫生 / 50w` 等).

### 步骤
1. 在 `/m/meetings/new` (跟上传同一会话 / 同一 draftId), 填:
   - 会议标题 `Q1 投诉评估`
   - 类型 hybrid 或 auto
   - brief: `Q1 物业投诉同比 +35%, 想拆原因 + 拟 Q2 整改, 预算 ≤ 50w`
2. 看 `✨ 让 AI 拆议程` 按钮 的 内文.
3. 点该按钮, devtools Network 看 POST `/api/meetings/decompose-agenda`.

### Pass
- 步 2 按钮文案含 `(用上 N 份资料)`, N 是 attachments 数 (≥ 1).
- 步 3 request body 含 `"client_draft_id": "<draft uuid>"`.
- response 200, items 数 2-6. 至少 1 个 item 的 `title` 或 `note` 出现
  attachment 数据关键词:
    `1287` / `+35%` / `+58%` / `1287 件` /
    `安保 542` / `卫生 387` / `莲花山` / `50w` / `6月30日` 等.
  (任 1 命中 都算 PASS)
- 命中 ≥ 2 个 关键词 → GREEN-PLUS.

### Fail
- response items 完全 没引用 attachment 数据 → orchestrator decompose 没拼上下文.
- 按钮文案 没显 "(用上 N 份资料)" → 前端 attachmentCount 没 wire.

### 必须复述
- POST decompose-agenda request body 原文.
- response body 完整 items 数组 JSON 原文 (不省略).
- 标出 命中 attachment 数据的字段 (用 `[命中:数据]` 注).

- 每 item `meeting_id` 等于新会议 id (字段 不再 null).
