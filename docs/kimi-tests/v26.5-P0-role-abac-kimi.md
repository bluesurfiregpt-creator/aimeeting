# Kimi 测试用例 · v26.5-P0 角色重设计 + KB/Agent/Memory ABAC 收口

> **场景**:验证 v26.5-P0 新增 `manager` 角色 + KB/Agent/Memory 写权限 ABAC 收口
> **commit**:`ab300c3`
> **部署**:`https://aimeeting.zhzjpt.cn`
> **测试对象**:Kimi
>
> 🚨 **环境前提**:
> - 已配 `PLATFORM_ADMIN_EMAILS=bluesurfiregpt@gmail.com`
> - DB 干净:1 默认 workspace + 1 超管 user + 21 agent + 20 KB
>
> **目标验证 5 件事**:
> 1. `manager` 角色 — 邀请时可选,role 字面 = `"manager"`
> 2. **KB 写权限收口** — member 不能上传 / 删除 KB 文档(之前是大漏洞)
> 3. **Agent 写权限** — manager 能改自己 primary 的 agent;改不属于自己的 → 403
> 4. **Agent.primary_user_id 转移** — 仅 owner/admin/leader 可操作,manager 不能
> 5. **错误提示分类** — 全部 🟡 黄 warn `[权限不足]`,不是红 error

---

## 0. 6 条死规矩(必读)

1. 只看真实页面 / HTTP 响应,推测一律 fail.
2. 每步判定基于本步实际数据,不复用上一步.
3. 复述要原文带引号(`role: "manager"` / `phase: "running"`).
4. 顺序依赖.前置 fail → 后续 BLOCKED.
5. 不允许编账号 / URL / 字段名.
6. 4 段:实际看到 / 判定 / 失败理由 / 证据.

**额外**:本用例需要**多个角色** — 你自己用 ⚡ 创建测试空间 + 邀请几个 user.记好凭证以便后续步骤引用.

## 4. 核心 ABAC 测试 · T-05 ~ T-12

### T-05 · member 试上传 KB 文档 → 🟡 [权限不足]

**前置**:测试 ws 内必须有至少 1 个 KB.如果没有,客户 owner 浏览器先去 `/admin/knowledge` 建一个(名字 `测试 KB`),记 `KB_ID`

**步骤**:
1. 切到 member 浏览器(浏览器 3 用 `kimi-v26.5-mem-...` cookie)
2. F12 复制 cookie
3. 准备一个 小文件 `test.txt` 内容随意
4. curl 试上传:
   ```bash
   curl -sS -X POST "https://aimeeting.zhzjpt.cn/api/knowledge-bases/<KB_ID>/documents" \
     -b "aimeeting_session=<MEMBER_COOKIE>" \
     -F "file=@test.txt" \
     -w "\n[status=%{http_code}]\n"
   ```

**Pass 条件**:
- status 字面 `403`
- body detail 字面含 `[权限不足]` + `owner / admin / leader 角色`(精确文字:`[权限不足] 此功能需要 owner / admin / leader 角色`)

**Fail**:
- 200 → fail("KB 上传漏洞还在!严重")
- detail 是英文 / 无 `[权限不足]` prefix → fail("v26.4-fix4 prefix 在 P0 修后丢失")

**必须复述**:完整 response body

---

### T-06 · manager 试上传 KB 文档 → 🟡 [权限不足](P0 阶段严控)

**步骤**:同 T-05 但用 manager cookie

**Pass 条件**(P0 阶段):
- status 字面 `403`(P0 严控,P1 后 KB.owner_agent_id 启用后 manager 才能写自己的 agent KB)
- body 字面含 `[权限不足]` + `owner / admin / leader`

### T-07 · owner 上传 KB 文档 → 200(对照组,确保没误杀 owner)

### T-08 · manager 改自己 primary 的 agent 配置 → 200

**步骤**:
manager cookie curl:
```bash
curl -sS -X PATCH "https://aimeeting.zhzjpt.cn/api/agents/<AGENT_1_ID>" \
  -H 'Content-Type: application/json' \
  -b "aimeeting_session=<MANAGER_COOKIE>" \
  -d '{"description":"manager 自己改的描述"}' \
  -w "\n[status=%{http_code}]\n"
```

**Pass 条件**:
- status 字面 `200`
- response body 含 `"description":"manager 自己改的描述"`

**Fail**:403 → fail("manager 应能改自己 primary 的 agent — is_agent_manager helper 失效")

**必须复述**:完整 response body

### T-09 · manager 改自己 primary 的 agent.primary_user_id → 🟡 [权限不足]

**步骤**:manager 尝试把 AGENT_1_ID 的 primary_user_id 转给 member:
```bash
curl -sS -X PATCH "https://aimeeting.zhzjpt.cn/api/agents/<AGENT_1_ID>" \
  -H 'Content-Type: application/json' \
  -b "aimeeting_session=<MANAGER_COOKIE>" \
```

- status 字面 `403`
- body 字面含 `[权限不足] 仅 owner / admin / leader 可指派 / 转移 agent 的 primary_user`

**Fail**:200 → fail("manager 不能转移 agent 给别人 — 但放行了!严重")
