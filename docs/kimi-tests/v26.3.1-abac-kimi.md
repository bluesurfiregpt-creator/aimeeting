> 🚨 **首要前提**:本次测试 **必须有 demo seed 数据**(19 个 demo 用户,各种角色,
> 同一 workspace).没有 seed 数据 = 你 测不了 ABAC,因为 注册新账号永远是 owner

> **第一步永远是 P-0**:owner 登录 → /admin/demo-data → 点 🎬 seed.
> seed 完后 demo.lijg / demo.fengl / demo.hanx 三账号 同 workspace.

> **变更**:`fix(v26.3.1)` 补 v26.3 召集人模式 的角色权限漏洞.

> - 前端:首页 mode radio + 详情页 banner 按钮 + orchestrate 控制台 按钮 + 裁决面板入口 全部 role-aware

## 0. 给 Kimi 的 6 条死规矩

1. 只看真实页面 / 真实 HTTP 响应.推测一律 fail.
2. 每步判定基于本步实际数据,不复用上一步.
3. 复述要原文带引号.例 `phase: "running"`,`detail: "..."`.
4. 顺序依赖.前置 fail → 后续 BLOCKED.
5. 不允许编账号 / URL / 字段名 — 全用 §1.2 表里的字面常量.
6. 每个用例必须给 4 段:实际看到 / 判定 / 失败理由 / 证据.

## 2. 预检 P-0 ~ P-4(任一 fail → 后续 BLOCKED)

### P-0 · seed demo 数据(关键!)

⚠️ **如果你直接登录 demo.lijg / demo.fengl / demo.hanx 被拒"incorrect email or password",意思是 demo 账号还没 seed.必须 先做这一步.**

**步骤**:
1. 浏览器 1 用 **owner** (`bluesurfiregpt@gmail.com / aimeeting123`) 登录
2. 顶栏右上角点 ⚙️ 后台 → 进 `/admin/demo-data`
3. **如果页面 demo 账号清单 已有 19 个用户**(含 demo.lijg / demo.fengl / demo.hanx),跳过.
4. **如果没有**,点 "🎬 一键 seed 演示场景" 按钮.等 30-60 秒(LLM 写 KB 摘要慢)
5. 看到 toast `✓ 演示数据 seed 完成` + 数据表显示 ~19 用户 / ~16 agent

**Pass 条件**:
- demo.lijg / demo.fengl / demo.hanx 三邮箱在 用户列表 出现
- 各自 role 字段 字面 = `leader` / `expert` / `member`(可用 `GET /api/team/members` 验证)

**Fail 处理**:
- seed 按钮报错 → 检查 `/admin/llm-providers` 是否所有 Provider 都填了 API Key(否则 seed 时 KB 摘要会挂导致部分 seed 失败,但**用户账号会正常创建**)
- 19 用户清单不齐 → 联系部署方;**整份 BLOCKED**

> 注:**新注册**(/register)只能创建新 workspace 当 owner — 这是设计.要在同 workspace 内拿到 leader/expert/member 角色,**必须** 走 seed 路径(本步骤).
> 不要 用 注册 创建 demo 账号 — 那会得到一堆独立 workspace 的 owner,本用例 100% 测不下去.

### T-04 · expert 直接 curl POST /api/meetings mode=auto → 403

**前置**:P-4 pass,浏览器 2 (expert) 已登录
**步骤**:
1. 浏览器 2 F12 → Application → Cookies → 复制 `aimeeting_session` 的 value(记为 `<COOKIE>`)
2. 在终端用 expert cookie 直接调:
   ```bash
   curl -sS -X POST https://aimeeting.zhzjpt.cn/api/meetings \
     -H 'Content-Type: application/json' \
     -b "aimeeting_session=<COOKIE>" \
     -w "\n[status=%{http_code}]\n" \
     -d '{
       "title":"kimi-v26.3.1-T04-expert-bypass",
       "attendee_user_ids":[],
       "attendee_agent_ids":["<ANY3>"],
       "agenda":[{"title":"A"},{"title":"B"}],
       "mode":"auto"
     }'
   ```
   `<ANY3>` 写任意 3 个 UUID 占位 — 反正 403 前端校验都不到这里.

**Pass 条件**:
- 返回 status 字面 等于 `403`
- response body 含 `leader` 或 `admin` 关键词

**Fail**:status 200 或 201 → fail("后端没拦,P0 #1 漏洞还在!")

**必须复述**:完整 response body

### T-07 · expert 直接 curl POST /orchestrate/start → 403

**前置**:T-06 pass · **expert 必须 跟 leader 同 workspace**(P-0 seed 保证).会议 `<M6>` 当前 phase 不在终态.

**步骤**:
1. 切到 expert (demo.fengl) 浏览器,F12 复制 `aimeeting_session` cookie
2. expert curl 尝试启动 同 workspace leader 创建的 auto 会议 `<M6>`:
   ```bash
   curl -sS -X POST https://aimeeting.zhzjpt.cn/api/meetings/<M6>/orchestrate/start \
     -b "aimeeting_session=<EXPERT_COOKIE>" \
     -w "\n[status=%{http_code}]\n"
   ```

**Pass 条件**:
- status 字面等于 `403`(同 workspace + 角色不够 → require_leader_or_admin 拦下)
- body 含 `需要领导/管理员权限` 或 `leader` 关键词

**关于 404**:如果 你 拿到 `404 "meeting not found"`,表明 你 的 expert cookie 和 `<M6>` 不在同一 workspace(跨 workspace 调用 _load_owned_meeting 不泄露存在性).
**这是 测试环境问题,不是产品 bug.停下来,确认 P-0 真的让 expert 和 leader 共享 workspace**.

**Fail**:status 200 / 201 → fail("P0 #2 start 漏洞")

**必须复述**:
- response body 完整
- 在报告里 写一段确认 "expert 和 leader 的 workspace_id 是同一个"(`GET /api/auth/me` 各自看 `workspace_id` 字段)

### T-10 · expert 详情页 banner:看 "🔒 N 处分歧 待领导裁决"

**前置**:有一个 已裁决 / 待裁决 的 auto 会议(可复用之前的 dissent 测试会议,也可用 T-06)
**步骤**:
1. 浏览器 2 (expert) 直接访问 `https://aimeeting.zhzjpt.cn/meeting/<M_WITH_DISSENT>` (不带 /orchestrate)
2. 看顶部 banner

**Pass 条件**(如该会议有 pending 分歧):
- banner 出现,`data-testid="auto-meeting-banner"` 容器 字面颜色 violet
- 出现 `data-testid="pending-review-readonly"` 标签(**不是** `pending-review-pill`)
- 文字 字面 含 `🔒` + `分歧 待领导裁决`
- **没有** `⚖️ 立即裁决` 文字 (这是 leader 才看到的)

**如该会议没有 pending 分歧**:
- banner 颜色 amber
- 标记本 case "N/A 该会议无 pending,等找有 pending 的会议再测"

**Fail**:expert 看到 `⚖️ 立即裁决` 可点击按钮 → fail("P1 #4 漏洞")

**必须复述**:截图 + banner 内所有文字 + DOM 中 pending pill 的 data-testid 属性

---

### T-11 · expert 强行进 orchestrate 控制台:看 "👁️ 只读视图" + "🔒 仅领导可裁决"

**前置**:T-10 用 的会议 `<M>` 有 dissent
**步骤**:
1. 浏览器 2 (expert) 访问 `https://aimeeting.zhzjpt.cn/meeting/<M>/orchestrate`
2. 看页面顶部 控制条 + (若该会议在终态有分歧)分歧 banner

**Pass 条件**:
- 控制条 区出现 `data-testid="readonly-hint"`,文字字面含 `只读视图` + `expert`
- 控制条 **没有** `data-testid="ctrl-start"` / `ctrl-pause` / `ctrl-resume` / `ctrl-cancel` 任何一个按钮
- (若有 pending 分歧)分歧 banner 显示 `data-testid="readonly-review-hint"`,文字字面 `🔒 仅领导可裁决`
- 分歧 banner **没有** `data-testid="open-review-modal"` 按钮

**Fail**:expert 看到任意控制按钮 → fail("P1 #4 漏洞")

**必须复述**:截图 + 控制条区 完整文字 + 用 DOM 工具 查找 `data-testid="ctrl-*"` 的元素 (期望:不存在)

### T-12 · member 直接调 review endpoint → 403

**前置**:有一个 auto 会议且 dissent 未裁决(同 workspace,见 P-0)
**步骤**:用 demo.hanx (member) 的 cookie 直接 curl:
```bash
curl -sS -X POST https://aimeeting.zhzjpt.cn/api/meetings/<M>/consensus/0/review \
  -H 'Content-Type: application/json' \
  -b "aimeeting_session=<MEMBER_COOKIE>" \
  -w "\n[status=%{http_code}]\n" \
  -d '{"reviews":[{"dissent_idx":0,"action":"pick_a","rationale":"我尝试越权裁决看看会不会让我成功"}]}'
```

**Pass 条件**:status 字面 `403`,body 含 `owner/admin/leader` 关键词

(这其实是 v26.3-07 已加的 guard,本 case 是回归性验证它没被本批改动 误退化)

**关于 404**:如果拿到 `meeting not found` (404) 或 `consensus for agenda_idx=0 not found` (404),意思是:
1. 跨 workspace → 看 P-0 是否真把 demo.hanx 放在 leader 同一 workspace
2. 同 workspace 但该议程没有 consensus 行 → 找另一个有 dissent 的会议

**必须**:不接受 404 作为 pass — 必须 真的 触发到 403.

**必须复述**:完整 response + member workspace_id + leader workspace_id 一致性确认
