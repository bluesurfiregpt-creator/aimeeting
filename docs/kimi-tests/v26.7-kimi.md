## A. Memory 检索一致性 (核心)

### T-01 · 多 AI 写 memory, 检索时 不串扰 ★
**前置**:
- 创建 2 个 AI: 房屋安全专家 (primary=MGR1), 物业管理专家 (primary=MGR2)
- MGR1 给 房屋安全专家 写 1 条 memory: "外墙脱落是 一级危险"
- MGR2 给 物业管理专家 写 1 条 memory: "电梯故障 24h 内修复"

**步骤**:
- 创建会议, 邀请 房屋安全专家 + MGR1 参会
- 在会议里 @房屋安全 提问: "我们小区的电梯有问题, 怎么办?"

**Pass**:
- AI 回答里 **不应** 引用物业的 memory ("电梯故障 24h 内修复")
- 因为 物业的 memory 通过 v26.7-01 agent_id 过滤被排除
- 后端 log (可选): "retrieve_relevant agent_id=房屋安全 ... 4 mems"
- 仅 房屋安全 自己的 memory + workspace 通用 memory 命中

### T-02 · workspace 通用 memory 仍然能被任何 AI 读到
**前置**: owner 写 一条 workspace 通用 memory (不挂任何 agent): "本工作空间是 福田区住建局"
**步骤**: 任何 AI 提问 "你在哪里工作" → AI 应能回答 "福田区住建局"
**Pass**: 通用 memory 被任何 agent 读到

## B. KB 沉淀目标一致性

### T-03 · task 沉淀 用 KB.owner_agent_id 反查 ★
**前置**:
- 创建 AI = 房屋安全专家, owner_agent_id 显式 指向新建 KB "房屋安全规范库"
- 该 AI 同时 `knowledge_base_ids = [其他 KB]` (不一致情况)

**步骤**: 派一个 task 给 该 AI → 办结 → curator approve

**Pass**:
- KB document 写入 "房屋安全规范库" (= owner_agent_id 反查的)
- 而不是 写入 `knowledge_base_ids[0]` (老逻辑)

### T-04 · 新建 KB 时自动设 owner_agent_id
**步骤**: task 沉淀给 1 个无 KB 的 agent → consolidator 自动新建 KB
**Pass**:
- 新建的 KB.owner_agent_id = agent.id (v26.7-02 新加)
- 不是 NULL

## D. 桑基图 sediment_pending + 节点编辑

### T-07 · 待审批草稿 显示 ⏳ 节点
**前置**:
- 制造 1 个 pending KbSedimentationDraft + 1 个 pending MemoryDraft

**步骤**: 进 /me/profile/lineage
**Pass**:
- 顶部 stat 卡 显示 "⏳ 待审草稿: 2"
- 桑基图有 2 个 ⏳ 节点 (label 前缀 "⏳ 待审 KB:" / "⏳ 待审 Memory:")
- 节点 → agent 边 颜色/透明度 跟正常边不一样 (sediment_pending kind)
- 来源 (任务/会议) → ⏳ 节点 也有连线

### T-08 · 节点 click → modal + 编辑入口 ★
**步骤**: 点 任意 memory 节点
**Pass**:
- 弹 modal, 显示 meta + 内容
- 底部 有 "✏️ 编辑" + "🗑️ 删除" 按钮 (memory 节点) 或 "✏️ 编辑" (kb_doc 节点)
- 点 "🗑️ 删除" → confirm → DELETE /api/memory/{id} → 列表刷新, 节点消失

### T-09 · KB doc 节点 编辑 → 跳 KB 详情页
**步骤**: 点 KB doc 节点 → "✏️ 编辑"
**Pass**: 显示 "进入 KB 编辑页" 链接, 点击跳 `/me/profile/knowledge/{kb_id}`
