## T-01 · 会议中途 刷新页面 不再 被 误判为 结束 ★ (fix1 核心)

**前置**: P-1 ~ P-3 全 pass.

**步骤**:
1. owner 进主页, 点 "新建会议", 题目 "v26.11-T01", 邀请 ≥ 1 个 AI, mode = hybrid, 提交
2. 跳进 会议室 (`/meeting/<id>`), 点 "▶ 开始会议", 允许 麦克风 权限
3. 等 顶部 状态 显示 "已连接, 开始说话" 或类似 (确认 WS 已通)
4. **不点 结束会议**, 直接 浏览器 刷新 (F5 / Cmd+R)
5. 重新 进 同一个 会议室 URL

**Pass** (必须 全部 满足):
- (a) 刷新后 会议室 仍然 打开 (不是 跳走)
- (b) 顶部 状态文字 = `"会议进行中,点击 🎙️ 恢复录音 继续"` (字面常量, 必须 一字不差)
- (c) "▶ 开始会议" 按钮 = **disabled (灰)** — 不是 可点击
- (d) 出现 一个 绿色 "🎙️ 恢复录音" 按钮 (`bg-emerald-500`, 文字 包含 麦克风 emoji)
- (e) "■ 结束会议" 按钮 = enabled (红色 边框, 可点击)
- (f) 后端 GET `/api/meetings/<id>` 返回 JSON, `status` 字段 **必须** 是 `"ongoing"` (不是 `"finished"`)

**Fail 情景**:
- 状态文字 是 "已结束" / "✅ 已处理" → bug 没修
- "▶ 开始会议" 是 enabled → phase 没识别成 live
- 没出现 "🎙️ 恢复录音" 按钮 → 重进逻辑 没接上
- GET 返回 `status: "finished"` → main.py 的 WS close 还在 偷偷 写 finished

**复述**:
- 截 顶部 4 个按钮 的 截图
- 贴 `curl -s -b "<cookie>" https://aimeeting.zhzjpt.cn/api/meetings/<id>` 的 完整 JSON 原文

## T-02 · 点 "🎙️ 恢复录音" 真的 重连了 麦克风 + WS

**前置**: T-01 pass.

**步骤**:
1. 继续 T-01 的 会议室 (status=ongoing 状态)
2. 点 "🎙️ 恢复录音" 按钮 (绿色)
3. 等 状态 文字 变化

**Pass**:
- (a) 状态 文字 变 为 `"🎙️ 录音已恢复"` 或 `"已连接, 开始说话"` (二选一, 看 backend 顺序)
- (b) "🎙️ 恢复录音" 按钮 消失 (因为 socketRef 不再 null)
- (c) 浏览器 DevTools → Network → WS → 看到 一条 新的 `ws://.../ws/stt?meeting_id=<id>` 连接, state = open

**Fail**: 状态没变 / 按钮没消失 / 没新 WS → resume 函数 没生效

## T-04 · 邀请 AI 弹窗 — 仅 owner / leader / 创建人 可见 ★ (fix2 ABAC)

**前置**:
- T-03 之后, 用 owner 重新 新建 一场 hybrid 会议 "v26.11-T04", 邀请 1 个 AI
- owner 先 点 "▶ 开始会议" 启 WS

**步骤**:
1. owner 在 会议室 中, AI 画廊 区 找到 "+ 邀请 AI" 按钮 (虚线 边框, 文字 包含 "+ 邀请 AI" 4 字)
2. 点击 → 应该 弹出 modal (有 标题 "邀请 AI 加入本场会议")
3. 关掉 modal
4. 浏览器 切到 隐身 窗口 / 另一 device, 用 **member** (`demo.hanx@...`) 登录
5. member 直接 访问 owner 的 会议室 URL (复制 URL)
6. member 看 AI 画廊 — 找 "+ 邀请 AI" 按钮 → 点击
7. 准备 调 后端 模拟 (member 试图 邀请 一个 AI):
   ```bash
   curl -sS -X POST -b "<member_cookie>" \
     -H "Content-Type: application/json" \
     -d '{"agent_ids":["<some-agent-uuid>"]}' \
     https://aimeeting.zhzjpt.cn/api/meetings/<id>/agents
   ```

**Pass**:
- (a) owner 点 + 邀请 AI → modal 弹出, 标题 = `"邀请 AI 加入本场会议"` (字面)
- (b) member curl POST → HTTP `403`, body 包含 `"[权限不足] 仅 会议创建人 + leader / admin / owner 可邀请 AI"` (字面)
- (c) owner curl POST (用 owner cookie) → HTTP `200`, body 含 `added`, `already_invited`, `invalid`, `attendee_agent_ids` 4 个字段

**Fail**:
- member 也成功 200 → ABAC 没生效, 任意 member 都能 改 房间
- owner 403 → ABAC 误伤 (创建人 应该 pass)

## T-05 · 邀请 AI 写入 attendee + 广播 给 所有 WS ★ (fix2 核心)

**前置**: T-04 (a) (c) pass, owner 在 会议室 A, WS 已连.

**步骤**:
1. owner 浏览器: 会议室 A, WS open (T-04 的会议)
2. owner 第二个 标签页 (或第二个浏览器): 同 owner 账号, 同一个 会议室 A URL, **不 启 录音** (即 status=ongoing, phase=live, 但 我们 不点 start — 仅 让 它 也 register WS 注册到 房间)
   - 注意: 第二个 标签 看起来 应该 自动 phase=live + 出现 "🎙️ 恢复录音" (T-01 验证过)
   - 第二个 标签 不需要 点 恢复录音; 这里 的 目的 是 让 它 也 注册一个 WS client. 改: **第二个 标签 点 "🎙️ 恢复录音"** 让它 注册 WS
3. 在第一个 标签 (owner, 主) 点 "+ 邀请 AI" → modal 弹出
4. 勾选 1 个 之前 没邀请的 AI → 点 "确认邀请"
5. 等 3 秒

**Pass**:
- (a) 第一个 标签 modal 关闭, 顶部 出现 toast `"已邀请 1 位 AI 加入会议"`
- (b) 第一个 标签 的 AI 画廊 立刻 出现 这个 新 AI 的 头像 + 名字 (无需 刷新)
- (c) 第二个 标签 (同 owner) 也 收到 toast `"已邀请 1 位 AI 加入会议"`, AI 画廊 同步出现 新头像
- (d) 后端 curl GET `/api/meetings/<id>` → `attendee_agent_ids` 数组里 含 这个 新 AI 的 uuid

**Fail**:
- 第一个 标签 modal 关掉 但 画廊 没刷新 → 自己 没听 WS event
- 第二个 标签 完全 没动静 → 广播 没工作 (broadcast 函数 没遍历 clients)
- DB 没写入 → endpoint 写 attendee 失败

**复述**:
- 贴 后端 GET meeting JSON 中 `attendee_agent_ids` 数组 原文 (含 新 AI uuid)
- 截 两个 标签 的 截图, 都 显示 新 AI 的 头像

## T-06 · 邀请 已在 会议中 的 AI → 幂等 跳过 (idempotent)

**前置**: T-05 pass (会议 A 已经 邀请 了 至少 2 个 AI).

**步骤**:
1. owner 直接 curl 后端:
   ```bash
   curl -sS -X POST -b "<owner_cookie>" \
     -H "Content-Type: application/json" \
     -d '{"agent_ids":["<已在会议中的-agent-uuid>"]}' \
     https://aimeeting.zhzjpt.cn/api/meetings/<id>/agents
   ```

**Pass**:
- HTTP 200
- JSON 中 `added` = `[]` (空数组)
- JSON 中 `already_invited` 数组 含 那个 agent uuid
- `attendee_agent_ids` 数组 没 多出 重复项 (相同 agent uuid 只出现 1 次)

**Fail**:
- `added` 含 重复 uuid → MeetingAttendee 表 多了 一行 重复 (违反 设计)
- HTTP 500 → endpoint 没处理 重复
