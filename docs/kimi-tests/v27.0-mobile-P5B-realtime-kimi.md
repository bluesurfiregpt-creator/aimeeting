1. 只看真实页面 / 真实 HTTP 响应.
2. 每步判定必须基于本步实际看到的数据.

### 数据库前置假设
- 至少 1 个 ongoing meeting + 邀请至少 1 个 AI 专家 (T-04 召 AI 测试用).
- 至少 1 个 ongoing meeting 设了 agenda (T-05/06 议程事件可触发).

## 预检 P-1 ~ P-3

### P-1 · WebSocket 端点可达
- 在 mobile 浏览器 devtools Network → WS panel, 进 ongoing meeting 详情页.
- **Pass**: 看到一条 `wss://aimeeting.zhzjpt.cn/ws/stt?meeting_id=<id>` 建立 (101 Switching Protocols).
- **Fail**: 没建立 / 401 / 4xx.

### P-2 · 顶部状态点 — 连接绿点
- 在转录折叠区展开后 (P5A 入口), meta 行最左边应有个状态点.
- 刚加载: 灰点 (idle).
- 连上后 (~1 秒内): 绿色脉动点.
- **Pass**: 绿色脉动点出现.

### P-3 · 转录折叠未展开时 WS 也开
- 不展开折叠区, 看 Network → WS panel.
- 即使不展开转录, 也应有一条 ws 连接 (因为 banner 需要 agenda 事件, page-level WS).
- **Pass**: 一条 ws 连接 active, 即使折叠没展开.
- **Fail**: 没 ws (说明 WS 只在转录里开, 错位).

## T-01 · transcript_persisted payload 含 text + speaker_name

### 前置 P-* PASS.

### 步骤
- 在 devtools Network WS panel, 点该 ws 看 messages 流.
- 等真人 final 一句, 应出现一条 `transcript_persisted` 消息.

### Pass
- 消息 JSON 字段含: `type: "transcript_persisted"`, `line_id`, `start_ms`, `end_ms`,
  **新加字段**: `text` (字符串), `speaker_name` (字符串 or null), `speaker_status` (字符串 or null).
- 缺老字段中任一 → fail.

### Skip
- 测试期间没真人发言 → SKIP (但 T-04 召 AI 时仍能验 agent_message_*).

### 必须复述
- WS message 完整 JSON 原文 copy-paste.

## T-02 · 转录 live append (mobile viewer, 不需自己录音)

### 前置
- 桌面端打开同一场 ongoing meeting + 开始录音 (其他人讲话).
- mobile 同时打开该 meeting 详情页 + 展开"完整转录"折叠.

### 步骤
- 录音端讲一句话, 等 ASR final.
- mobile 端转录列表底部应自动追加一条新行 (灰底, 时间 + speaker + text).

### Pass
- 新行出现在底部, text 跟桌面端 final 一致.
- meta 行 "共 N 句真人" 数字 +1.
- 自动滚到底 (用户没主动上滑前提下).

### Skip
- 没第二台设备 (没人讲话). 用 T-04 召 AI 替代.

### 必须复述
- mobile 端 transcript 截图 (录音前 vs 录音后).

## T-03 · agent_message streaming — AI 一字一字吐

### 前置 P-* PASS, 该 meeting 有 attending_agents.

### 步骤
1. 在 mobile 详情页, 点底部 sticky "💬 召 AI" → 弹 sheet → 选一个 AI → "派他发言".
2. toast 出现: "已请 X 发言, 转录区可看实时打字".
3. 立刻展开"完整转录"折叠.
4. 几秒内, 转录列表底部追加一条紫色 AI 行:
   - 顶部 header: ◆ 色块 + agent nickname + 紫色 chip "正在打字…"
   - 文本初始空, 然后**字一段一段地添加** (chunk 追加)
   - 文本末尾有个紫色脉动光标 |
5. AI 完成发言后:
   - "正在打字…" chip 消失
   - 光标消失
   - 文本是最终完整版

### Pass (全部)
- 紫色 AI 行出现, 不是 4 秒后整段刷出来的, 而是 chunk by chunk.
- "正在打字…" chip + 光标在 streaming 期间显示.
- 完成后 chip + 光标都消失.

### Fail
- AI 行 4 秒后整段才出现 (说明 streaming 没生效, 还在用老 setTimeout reload).
- 光标永远不消失 (说明 agent_message_end 没收到).

### 必须复述
- 录屏或 3 张截图 (streaming 进行中, streaming 完成, header chip 区别).

## T-04 · agent_message_chunk WS 事件 raw 检查

### 前置 T-03 PASS.

### 步骤
- 在 T-03 召 AI 期间, devtools WS messages panel 看应有的事件序列:
  1. `agent_message_start` 一条 — `agent_id`, `agent_name`, `agent_nickname`, `agent_color`
  2. 多条 `agent_message_chunk` — 每条 `agent_id` + `chunk` (字符串)
  3. `agent_message_end` 一条 — `agent_id`, `text` (完整最终), `citations` 数组 (可能为空)

### Pass
- 三种事件按顺序出现.
- chunk 数 ≥ 1 条.
- end 事件的 `text` 字段非空, 跟 mobile 显示的最终文本一致.

### 必须复述
- WS messages 原始 JSON (start + 任一 chunk + end 三条).

## T-05 · agenda_advanced WS 静默 reload

### 前置
- 有 controller 权限 + 一场 ongoing meeting + agenda.
- 推荐两个端 (mobile + 桌面同时打开).

### 步骤
1. 在 mobile 端进 meeting 详情, 看顶部 StageChipsRow 当前 active 项是哪个 (例如 idx=1).
2. 在**桌面端** (或另一 mobile tab) 点 "推进议程".
3. 观察 mobile 端 (没主动点):
   - StageChipsRow active 项自动右移一格 (idx=2).
   - CurrentTopicCard 标题切到下一议程项.
   - 若议程被推到完结, 主区域显示 "议程已全部完成" 绿提示 + 顶部 toast "议程已全部走完".

### Pass
- mobile 端**没刷新页面**, agenda 自动同步.
- 主区域内容跟着切换.

### Fail
- 必须手动刷新才能看见进度.
- chip 没动.

### Skip
- 没法测多端, 标 SKIP.

## T-06 · 议程横幅 banner (off_topic / time_warning / stuck)

### 前置 P-* PASS. 该 meeting 有 agenda + 真人正在讨论 (orchestrator agenda_monitor 才会工作).

### 步骤
- 等真人讨论持续偏题 (5+ 句不沾边), 或议题时间用满 80%+, 或重复立场陷入循环.
- 这时 backend orchestrator (agenda_monitor.py) 会推:
  - `agenda_off_topic` → mobile 顶部 sticky 显黄色横幅 "🧭 议题似乎跑偏了" + body 描述
  - 或 `agenda_time_warning` → "⏰ 时间快用完 (已 N 分钟)"
  - 或 `agenda_stuck` → "🔄 议题卡住了"
- 横幅右侧有 × 关闭按钮.

### Pass
- 横幅出现.
- emoji + 标题 + body 跟事件 payload 字段对应.
- 60s 后自动消失, 或点 × 立即消失.

### Skip 允许
- 测试期间没触发任何议程事件 (orchestrator 是被动监控). SKIP + 注明 "没机会触发".

### 必须复述
- 横幅截图 (若出现).
- WS panel 对应事件 JSON.

## T-07 · 断网重连

### 前置 P-* PASS.

### 步骤
1. 在 mobile 端展开转录, 看绿点.
2. 在 devtools Application → Service Worker 或 Network panel → throttling 切 "Offline".
3. 几秒内绿点应变 **黄色脉动** (重连中).
4. 切回 "No throttling".
5. 几秒内绿点应回 **绿色脉动** (已重连).
6. 重连后内部应自动 reload 一次拿可能错过的行 (Network 看到一个 `GET /api/m/meetings/<id>/transcript` 请求).

### Pass
- 状态点反映连接生命周期 (绿 → 黄 → 绿).
- 重连后内部 fetch transcript.

### Fail
- 状态点一直绿, 没反映断网.
- 重连失败 (黄点变红色) — 但 throttling 是临时, 应该能重连.

## T-08 · 桌面端旧行为没坏

### 前置 P-* PASS.

### 步骤
- 用桌面端浏览器 (非 mobile UA), 进 desktop /meeting/<id> 旧页面.
- 录音, 召 AI.

### Pass
- 桌面端 transcript 接收正常 (老 interim + final + persisted 流程).
- 桌面端 AI 召唤后 streaming 显示正常.
- 桌面端没异常报错 (devtools Console 干净).

### Fail
- 桌面端出现错误 (说明后端 broadcast 改动破坏了 desktop).

## 反幻觉自检

- "AI streaming 一字一字吐" 必须录屏或多张时间序列截图佐证, 不能只一张终态截图.
