1. 只看真实页面 / 真实 HTTP 响应.
2. 每步判定必须基于本步实际看到的数据.
3. 复述要原文 (带引号的精确字符串).
4. 不允许跳步.

### P-2 · 登录 + 进有转录的 meeting + 看到折叠区

  - 文案 "▾ 完整转录 (N 句)" (N 是 transcript_total)

## T-01 · 展开 → lazy load + 不展开不加载

### 前置 P-* PASS.

### 步骤
1. 打开 devtools Network panel, 清空.
2. 进 /m/meetings/<id>.
3. **不展开** 折叠条, 观察 Network: 应该看不到 `/api/m/meetings/<id>/transcript` 请求.
4. 点折叠条展开.
5. 文案变 "▴ 完整转录 (N 句)" (向下箭头变向上).
6. Network 出现一次 `/api/m/meetings/<id>/transcript` 请求, 200.
7. 折叠区下方显示真组件 (不是占位文字 "Phase 2 — 完整转录视图").

### Pass (全部)
- 不展开时, 没 transcript API 请求 (lazy load 生效).
- 展开后, 1 个 transcript API 请求.
- 旧占位文案 "Phase 2 — 完整转录视图" **不出现**.
- 展开后看到 meta 行 "共 N 句真人 · M 条 AI 发言" + 右侧 "↻ 刷新" 按钮 + 下方列表.

### Fail
- 一进页面就拉了 transcript (说明没 lazy).
- 仍显示 "Phase 2" 占位.
- 拉了但渲染失败.

### 必须复述
- Network 截图 (清空 → 展开前 → 展开后, 三个状态).
- 折叠区截图含 meta 行.

## T-02 · API response 字段 shape

### 前置 T-01 PASS, transcript 已展开过.

### 步骤
- devtools Network 找 `/api/m/meetings/<id>/transcript` 的 response.
- 检查 top-level 字段: `meeting_id, title, status, started_at, total_user_lines, total_agent_lines, lines`.
- 检查 lines[0] 字段: `kind ∈ {"user","agent"}, id, text, at_minute, created_at, speaker_name?, speaker_status?, agent_id?, agent_name?, agent_nickname?, agent_color?, trigger?, citations_count`.

### Pass
- 所有 top-level 字段存在.
- 每条 line 有 kind 字段, 值是 "user" 或 "agent".
- lines 按 created_at 正序 (检查相邻两条 created_at 升序排列).
- user 类: speaker_name 或者 null, agent_* 字段都是 null.
- agent 类: agent_name/agent_id 非空, speaker_* 都是 null.

### Fail
- 任一字段缺失.
- kind 出现其他值.
- 顺序错乱.

### 必须复述
- response 完整 JSON 第一条 line + 最后一条 line.

## T-03 · 真人发言行渲染

### 前置 T-01 PASS.

### 步骤
- 在展开的转录列表里, 找一条真人发言行 (kind=user).
- 视觉: 左侧时间 (例如 "12m" 或 "1h5m") + 说话人名字 + 文本.
- 背景: ink-900/40 (灰底, 不像 AI 行那么紫).

### Pass
- 时间格式正确: < 60 分用 "Xm" (无小时), >= 60 分用 "XhYm" (Y=0 时只 "Xh").
- 说话人名字 = JSON `speaker_name` 字段 (若 null 显 "未识别" 灰色).
- 文本 = JSON `text` (不截断).

### Fail
- 时间显示 0m 但 line 实际 at_minute > 0.
- speaker_name null 时没有 fallback "未识别".

### 必须复述
- 截图 1 条真人行.
- 该行对应 JSON.

## T-04 · AI 发言行渲染

### 前置 T-01 PASS, 该 meeting 有至少 1 条 AI 发言.

### 步骤
- 在转录列表里找一条 AI 行 (kind=agent).
- 视觉: 紫色边框 + 浅紫底, 顶部 header:
  - 左侧色块圆头 ◆ (颜色 = agent_color, 比如 violet/emerald/...)
  - nickname/name
  - trigger chip (例: "召唤" / "自动" / "关键词" / "@") — 若 trigger 字段非空
  - 右侧时间 "Xm"
- 主体: 文本 (15px).
- 底部: 若 citations_count > 0, 显 "📎 引用 N 条 KB".

### Pass
- 色块颜色匹配 agent_color (例: agent_color="violet" → bg-violet-500 紫色圆头).
- 名字显示优先 nickname, 否则 name.
- trigger 中文 label 对应正确 (manual="召唤" / auto_orchestrator="自动" / keyword="关键词" / at_mention="@").
- 文本不截断.

### Fail
- agent 行用了真人行的视觉 (没有紫色边框).
- trigger 显示英文原值 "manual" 不是中文.

### Skip
- meeting 内无 agent 行, SKIP. (检 JSON `total_agent_lines==0` 时.)

### 必须复述
- 截图 1 条 AI 行.
- 该行对应 JSON.

## T-05 · 时间正序排列 (真人 + AI 合并)

### 前置 T-01 PASS, 该 meeting 既有 user 行又有 agent 行.

### 步骤
- 看列表前 5 行 (或全部, 若少于 5 行), 记下每行 created_at + kind.
- 检查相邻两行 created_at 单调递增.
- 检查列表里 user 行 + agent 行交错出现 (不是 user 全在前面 agent 全在后面).

### Pass
- 全列表 created_at 升序.
- 至少有一次 user → agent 或 agent → user 切换 (说明真的按时间合并了).

### Fail
- user 行集中在前, agent 行集中在后 (说明没合并按时间排).
- 时间逆序.

### Skip
- 整个会议只有 user 或只有 agent, SKIP.

### 必须复述
- 前 5 行的 created_at + kind 列表 (顺序写出).

## T-06 · 刷新按钮

### 前置 T-01 PASS.

### 步骤
1. 在折叠区展开状态下, 看右上角应有 **"↻ 刷新"** 按钮 (32px 高灰边按钮).
2. 点之.
3. 按钮文案变 **"刷新中…"** disabled.
4. devtools Network 看一个新的 `/api/m/meetings/<id>/transcript` 请求.
5. 完成后按钮变回 "↻ 刷新".

### Pass
- 点击后发请求.
- 按钮 loading 状态正确.
- 数据可能跟之前一样 (内容没变化), 但元数据计数应跟新 (`total_user_lines` 等可能 +N 若期间真有新行写入).

### Fail
- 点了没反应.
- 按钮 loading 状态没变.

### 必须复述
- Network 截图.

## T-07 · 空态 / 无 transcript_total=0 不显示折叠区

### 前置
- 找一个 transcript_total == 0 的 meeting (从 /api/m/workbench 或 /m/meetings).

### 步骤
- 进该 meeting 详情页.
- 看是否还有 "完整转录" 折叠区.

- 进 meeting 仍能看到其他区 (Hero, Stage, CurrentTopic 等).

### Skip
- 找不到 transcript_total=0 的 meeting (workspace 内所有会议都有发言).
