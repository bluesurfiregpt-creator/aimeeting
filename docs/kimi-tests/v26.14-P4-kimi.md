# 🟦 P4.2 用例 (T-05 ~ T-06)

> P4.2 是 纯 后端 改 — agenda_monitor 加 过滤 + 三级 schema. 测试 走 `POST /api/meetings/{id}/agenda-monitor/run-now` dev endpoint (force=true 跳 cooldown).
> Kimi 需 用 owner cookie. cookie 名 = `aimeeting_session`.

## T-05 · 后端 LLM 返 严重度 字段 (schema 已 升 三级) ★

**前置**: P-3 pass + 有 一个 **进行中 (status=ongoing) + 有 agenda + 已 累计 ≥ 5 条 transcript** 的 会议 (用 `<MID>` 代 ID).

**步骤**:
```bash
# 用 owner cookie (从 浏览器 dev tools 复制 aimeeting_session)
COOKIE='aimeeting_session=...'
curl -sS -X POST "https://aimeeting.zhzjpt.cn/api/meetings/<MID>/agenda-monitor/run-now" \
  -H "Cookie: $COOKIE" | jq
```

**Pass** (response JSON 复述 必须 包含, 字段 字面 一致):
- (a) HTTP 200
- (b) 若 LLM 判 没事: response 含 `"fired_payload": null` (or `null`)
- (c) 若 LLM 判 有事 + 是 off_topic 类型: `fired_payload.type = "agenda_off_topic"` **AND** 至少 一个 下面 字段 出现:
  - `fired_payload.off_topic_severity` ∈ `["suspected", "confirmed", "severe"]`
  - OR `fired_payload.auto_summon_after_s` 为 数字 (severe 时)
- (d) 若 fire 了 type=agenda_stuck 或 time_warning: 老 schema, 不强求 severity

**判定 关键** (Kimi 重点 看):
- 不能 出现 `"off_topic": true` 但 **没** `"off_topic_severity"` 字段 — 那 说明 后端 没 deploy
- `"off_topic_severity": "none"` 不应 出现 在 fired_payload (那 时 应 不 fire, fired_payload=null)

**Fail 情景**:
- HTTP 500 → 后端 报错, 拉 logs
- 没 `off_topic_severity` 字段 出现 但 fire 了 off_topic → 后端 没 deploy
- response 复述 不出 时: 列 `curl -i` 完整 头 + 体

**复述**:
- 完整 JSON response (不允许 省略 字段)
- 若 fired_payload 非 null, 列 全部 字段 + 字面 值
