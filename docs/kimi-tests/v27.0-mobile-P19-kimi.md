## 预检 P-1 ~ P-4

### P-1 · 服务器 / 前端健康
- `curl -s -o /dev/null -w "%{http_code}" https://aimeeting.zhzjpt.cn/` → **必须 200**.
- `curl -s -o /dev/null -w "%{http_code}" https://aimeeting.zhzjpt.cn/m` → **必须 200**.

### P-2 · 后端 API 活
- `curl -s -o /dev/null -w "%{http_code}" https://aimeeting.zhzjpt.cn/api/meetings`
  → **必须 401** (未登录).
- `curl -s -o /dev/null -w "%{http_code}" -X POST -H "content-type: application/json"
   -d '{"brief":"smoke 测试 10 个字够长了"}' https://aimeeting.zhzjpt.cn/api/meetings/decompose-agenda`
  → **必须 401** (未登录, 不是 404).
- 任一 5xx 或 404 → P-2 FAIL, 后续 T-* 全 BLOCKED.

### P-3 · 登录 owner
- 浏览器 / chrome MCP 打开 `https://aimeeting.zhzjpt.cn/m`, 用 owner 账号登录.
- **Pass**: 跳到 `/m` 后顶部 PageHeader 右上角能看到自己头像 / 名字, 不再出现登录页.

### P-4 · 至少 3 个 active expert AI
- 进 `/m/meetings/new`, 滚到"邀请 AI 专家"区, 数 chip 数量.
- **Pass**: ≥ 3 个 chip 出现.
- **Fail**: < 3 个 → T-05 (auto 模式 创建) BLOCKED, 其他 T-* 仍可测.

任一 P-1 / P-2 / P-3 fail → 后续 T-* 全 BLOCKED.

## T-01 · 创建页 brief textarea 渲染 + 字数 counter

### 前置 P-1 ~ P-3 PASS.

### 目标
确认 P19 加的 `description` textarea 在创建页可见, 默认空, 字数 counter 工作.

### 步骤
1. 浏览器进 `https://aimeeting.zhzjpt.cn/m/meetings/new`.
2. 滚到"会议类型"下方, 应看到一节标题:`会议 brief / 诉求`.
3. mode 默认 `hybrid` 时, Label 后面缀字面常量 `· 选填` (灰色 text-zinc-500).
4. 切到 `auto` (点 "全 AI 自主讨论" radio), Label 后缀变成 `· 全 AI 模式 必填` (accent 色).
5. 在 textarea 输入 50 个汉字.
6. 看右上角字数 counter.

### Pass
- 步 2 看到"会议 brief / 诉求"section.
- 步 3 hybrid 模式: 文案 `· 选填` 字面命中.
- 步 4 auto 模式: 文案 `· 全 AI 模式 必填` 字面命中.
- 步 5 后 counter 显示 `50/2000` (50 是中文字符数, 不是字节).
- textarea border:
  - hybrid: 默认 ink-800 灰 border
  - auto: accent-500/40 (略带紫的高亮 border)

### Fail
- 步 2 section 找不到 → P19 没部署 / 部署回滚.
- counter 显示 `1/2000` 或不更新 (说明 onChange 没 wire).
- counter 显示 `150/2000` (说明用了 byte 长度而不是字符).

### 必须复述
- 步 5 textarea + counter 区域的截图.
- 步 3 + 步 4 Label 后缀文案 的原文.

## T-03 · brief ≥ 10 字时 显 AI 拆议程 按钮 + 端点接受请求

### 前置 T-01 PASS.

### 目标
端到端验 POST /api/meetings/decompose-agenda — 真调 LLM, 真返回 2-6 个议程项.

### 步骤
1. 在 brief textarea 输入以下原文 (60 字, 字面常量):
   ```
   Q1 物业投诉同比 +35%, 主要 集中 在 安保 / 卫生.
   想 拆原因 + 拟 整改 方案, Q2 落地, 预算 ≤ 50w.
   ```
2. 找到下方按钮 "✨ 让 AI 拆议程", 点击.
3. 按钮 文案变 "AI 拆议程中…" + 出现 spinner.
4. 等 5-30 秒.
5. 议程区原本只有一个空议程项, 现在应被替换 — 多个新议程项出现.
6. 同时屏幕底部弹 toast: `AI 已拆出 N 个议程项, 你可继续编辑` (N 是数字, 2-6).

### Pass
- 按钮点击后 立刻 显 "AI 拆议程中…" + spinner.
- 议程列表项数 ≥ 2 且 ≤ 6.
- 每个新议程项 title 非空, 中文.
- 每个新议程项 默认 expand 了"方向提示" textarea, 内含 30-80 字的 note (内容跟 brief 里的"投诉 / 安保 / 卫生 / 预算 / Q2"等关键词强相关).
- toast 文案匹配 `AI 已拆出 ` + 数字 + ` 个议程项, 你可继续编辑`.
- devtools Network → 看到一条 POST `/api/meetings/decompose-agenda` request → response 200, body 是
  `{"items":[{"title":"...","note":"...","time_budget_min":...}, ...]}` 数组.

### Fail
- 点击没反应 / 没 spinner → onClick 没 wire.
- 30 秒后还没出结果 → 后端 LLM provider 没配 / 超时.
- 返回 0 项 → LLM 输出不含 JSON 数组 (后端会 500). 看 Network response.
- 返回的 note 字段都是 `null` / 全空 → prompt 没引导出 note (可标观察, 不算 fail).
- 议程项是 "现状分析 / 未来展望 / 总结" 万能议程 → prompt 没起效, 标观察.

### 必须复述
- POST `/api/meetings/decompose-agenda` request body 原文.
- response body 完整 JSON 原文 (复制完整, 不省略).
- 议程区"AI 拆完后" 截图.

## T-04 · AI 拆议程时 现有议程有内容 → 弹 confirm

### 前置 T-01 PASS, T-03 之前 (议程未拆).

### 步骤
1. 刷新页面 (确保是默认空议程 + 0 字 brief).
2. 在唯一议程项 title 框 输入 `自填议程标题 ABC`.
3. brief textarea 输入 30 字以上 (例 `测试 AI 拆议程 confirm 弹窗 的 条件`).
4. 点 "✨ 让 AI 拆议程".

### Pass
- 出现 fixed inset-0 z-50 的全屏暗色覆盖层.
- 覆盖层中弹卡片 含 标题 `覆盖现有议程?`.
- body 文案: `你已填了一些议程项. 让 AI 拆议程会替换它们 — 已有内容会丢失.`
- 两按钮:`再想想` (左, 灰) + `覆盖` (右, 紫).
- 点 `再想想` → 卡片消失, 议程 `自填议程标题 ABC` 仍在.
- 重新点 "✨" → 重弹卡片 → 点 `覆盖` → 议程被 AI 拆出 的项替换.

### Fail
- 没弹 confirm 直接覆盖 → 数据丢失风险, MUST FIX.
- 弹的不是 fixed 覆盖层 而是 浏览器 alert → 用户体验 fail (mobile alert 丑且不可控).
- 文案不一致.

### 必须复述
- confirm 卡片截图.
- 复述卡片内 4 句文字 (标题 + body + 2 按钮).

## T-07 · description 落库 + 详情页 API 能取出来

### 前置 T-06 创建成功, 拿到 meeting_id.

### 步骤
1. 在 T-06 创建会议时 brief 也填 `这是 测试 描述 文本 用 来 验 落库 50 字 abc def 123`.
   (即 T-06 改成同时填 brief, 然后创建)

3. devtools Network 找一条 `GET /api/meetings/<id>` (note: 桌面 path 而非 /api/m/).
4. 看 response body.

### Pass
- response body JSON 含 `"description": "这是 测试 描述 文本 用 来 验 落库 50 字 abc def 123"`
  (原文不变, 末尾不带额外 trailing whitespace).

### Fail
- `description` 字段缺失 → MeetingOut schema 没暴露.
- `description: null` 但用户明明填了 → create_meeting 没 persist.
- 多了 trailing whitespace → 后端 strip 没 work.

### 必须复述
- response body 完整 JSON 原文 (至少 description / id / title 3 字段).

## T-08 · auto 模式跑通 → moderator intro 体现 brief / note

### 前置 P-4 PASS, T-03 PASS.

### 目标
端到端 auto 会议跑一遍, 看 LLM moderator 的 intro 里 真 引用 了 brief / note 的具体词
(不是泛泛而谈).

### 步骤
1. 进 `/m/meetings/new`. 填:
   - 标题: `auto 测 brief 落地`
   - 类型: `全 AI 自主讨论`
   - brief: `Q1 物业投诉 +35%, 主因 安保 / 卫生. 拟 Q2 落地 整改, 预算 ≤ 50w.`
   - 议程: 先填一个 placeholder 标题 + 时长, 然后点 "✨ 让 AI 拆议程"
     → 自动拆出 2-6 项.
   - AI 专家: 选 ≥ 3 个 expert.
2. 点 "创建会议". 跳到详情页. 应自动 start → ongoing.
3. 等 30-60 秒 (auto 会议 moderator intro 段落生成需要一两次 LLM call).
4. 展开"完整转录"折叠区 / 或 滚到议程下方 看 moderator intro 段.

### Pass
- moderator intro 文本里 出现至少 1 个 brief 里 出现过的关键词:
  `Q1` 或 `投诉` 或 `安保` 或 `卫生` 或 `Q2` 或 `50w` 或 `预算`.
- 议程拆出来若 含 note (Q3 时间窗 / 预算 之类), moderator 还可能 提到该约束.

### Fail
- intro 完全没引用 brief 词汇, 只说 "今天 我们 讨论 XX 议程 …" 那种空话
  → prompt 模板没起效 / 模板 string format 失败 (后端日志看 KeyError).
- 后端 500 / orchestrator 崩 → 看 backend logs.

### 跳过条件
- 没足够 expert AI / LLM provider 没配 → SKIP.

### 必须复述
- moderator intro 完整段落原文.
- 标出该段中 命中的 brief 关键词 (用 `[命中:词]` 标注).
