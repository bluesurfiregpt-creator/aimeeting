### T-WS-2:WS upgrade handshake 走 `101 Switching Protocols`(必须 在 T-03 之前!)

- **前置**:T-02 pass
- **目标**:验证 prod nginx → backend 的 WS upgrade 通路本身畅通。**必须在
  T-03 启动 orchestrator 之前先做** — 因为本 case 关心的是 WS 通路,跟有没有
  事件无关
- **步骤**:
  ```
  WSKEY=$(openssl rand -base64 16)
  curl -sS -i --max-time 5 \
    -H "Connection: Upgrade" \
    -H "Upgrade: websocket" \
    -H "Sec-WebSocket-Version: 13" \
    -H "Sec-WebSocket-Key: $WSKEY" \
    -H "Cookie: aimeeting_session=$JWT_TOKEN" \
    "https://aimeeting.zhzjpt.cn/ws/stt?meeting_id=<MEETING_ID>" \
    2>&1 | head -20
  ```
- **Pass 条件**:
  - 输出第一行含 `"HTTP/1.1 101"` 或 `"HTTP/2 101"`
  - 输出含 header `Upgrade: websocket`(大小写不敏感)
  - 输出含 header `Sec-WebSocket-Accept: ...`
- **Fail 条件**:
  - 输出第一行是 `HTTP/1.1 401` → cookie / 鉴权出错
  - 输出第一行是 `HTTP/1.1 400` / `403` → nginx / backend 拒
  - 输出第一行是 `HTTP/1.1 200` 普通响应 → nginx 没 upgrade(配置丢失)
  - timeout / 连接 reset
- **必须复述**:`head -20` 完整原文

### T-WS-1:**核心 — Python websockets 抓真实 frame** ⭐

- **前置**:T-03 pass(orchestrator 已 trigger,接下来 60s 内会 broadcast 多个事件)
- **目标**:**这是 Sprint 3 最关键的 case** — 直接抓 WS frame 看是否含
  `agent_message_start` / `agent_message_chunk` / `agent_message_end` /
  `orchestrate_phase_change` 这 4 个 type 字面值。这是区分 "WS 推真接通"
  vs "fallback 轮询兜底" 的唯一方法。Sprint 2-3 这条 case 必 fail(没 broadcast),
  Sprint 3 必 pass(broadcast 实装)
- **步骤**:
  1. 装库(如果还没装):`pip install --quiet websockets || python3 -m pip install --quiet websockets`
  2. 跑 capture 脚本 90 秒:
     ```
     python3 <<EOF > /tmp/kimi-ws-frames.log 2>&1
     import asyncio, json, sys, time
     import websockets

     URL = "wss://aimeeting.zhzjpt.cn/ws/stt?meeting_id=<MEETING_ID>&token=$JWT_TOKEN"
     CAPTURE_SEC = 90

     async def main():
         start = time.time()
         frame_count = 0
         type_counts = {}
         try:
             async with websockets.connect(URL, open_timeout=10) as ws:
                 print(f"[CONNECTED] {time.time() - start:.2f}s after start", flush=True)
                 while time.time() - start < CAPTURE_SEC:
                     try:
                         msg = await asyncio.wait_for(ws.recv(), timeout=CAPTURE_SEC - (time.time() - start))
                     except asyncio.TimeoutError:
                         break
                     frame_count += 1
                     try:
                         obj = json.loads(msg)
                         t = obj.get("type", "?")
                         type_counts[t] = type_counts.get(t, 0) + 1
                         print(f"[+{time.time()-start:6.2f}s] {msg[:300]}", flush=True)
                     except Exception:
                         print(f"[+{time.time()-start:6.2f}s] (non-json) {msg[:200]}", flush=True)
             print(f"[DONE] total_frames={frame_count} types={type_counts}", flush=True)
         except Exception as e:
             print(f"[ERROR] {type(e).__name__}: {e}", flush=True)

     asyncio.run(main())
     EOF
     cat /tmp/kimi-ws-frames.log
     echo "---SUMMARY---"
     tail -1 /tmp/kimi-ws-frames.log
     ```
- **Pass 条件**(**5 条全部满足**):
  1. log 第一行 含 `"[CONNECTED]"`(WS 接通)
  2. `type_counts` 含 key `"agent_message_start"`,值 ≥ 1
  3. `type_counts` 含 key `"agent_message_chunk"`,值 ≥ 1
  4. `type_counts` 含 key `"agent_message_end"`,值 ≥ 1
  5. `type_counts` 含 key `"orchestrate_phase_change"`,值 ≥ 1
- **Fail 条件**:
  - 出现 `[ERROR]` 行,且无 `[CONNECTED]` → WS 没接通
  - `[CONNECTED]` 出现但 `type_counts` 里 4 个 key **任何一个缺失或 = 0** →
    **Sprint 3 broadcast 未生效**(Kimi 这一步 fail 就是本测验的核心结论)
  - log 含 `system` type frame 且 `"msg":"auth required"` → JWT token 出错
- **必须复述**:
  - `/tmp/kimi-ws-frames.log` 的 **第一行**(`[CONNECTED]` 或 `[ERROR]`)
  - 含 `agent_message_start` 字符串的 **第一个 frame 完整 JSON 原文**
  - 含 `orchestrate_phase_change` 字符串的 **第一个 frame 完整 JSON 原文**
  - 最后一行 `[DONE] total_frames=N types={...}` 原文
- **备注**:本 case **优先级最高**,如果它 pass,即使 T-06 / T-07 因为浏览器
  环境问题(WS 不稳)看不到 UI 更新,Sprint 3 的服务端实现仍可判 GREEN

## 已知不在本次 Sprint 3 范围 (Kimi 见到这些不算 fail)

- **真 chunk-by-chunk streaming**:本次是 blob-style(start + 单 chunk = 完整
  text + end)。T-WS-1 抓到的 `agent_message_chunk` frame 的 `chunk` 字段值
  通常等于整段 reply,不是 字符级逐个推。Sprint 4 / 5 再上 真 streaming
- **`agent_recommendation` / `dissent_detected` / `agenda_off_topic`**:这些是
  hybrid 会议路径的事件,auto 会议里不出现,T-WS-1 看不到不算 fail
- **`transcript_persisted`**:这是真人发言 ASR 路径,auto 会议无真人发言,看不到正常
- **`reconnect_state`**:这是 sttSocket.ts 本地 emit,后端不发,Kimi 在 T-WS-1
  log 里看不到正常
- **P95 延迟数值**:Kimi 不精确测 ms 级延迟,本测验只验功能正确
- **iOS 微信 WKWebView 行为**:本测验在桌面浏览器跑,不覆盖小程序壳
