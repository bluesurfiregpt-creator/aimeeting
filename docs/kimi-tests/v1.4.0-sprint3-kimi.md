### P-2: 后端 API 活

- **步骤**:`curl -sS -o /dev/null -w "%{http_code}" https://aimeeting.zhzjpt.cn/api/auth/login`

- **步骤**:
  ```
  curl -sS -X POST https://aimeeting.zhzjpt.cn/api/auth/login \
    -H "Content-Type: application/json" \
    -d '{"email":"demo.lijg@futian.gov.cn","password":"demo123"}' \
  ```

## 测试用例 T-01 ~ T-08

### T-01:创建 mode=auto 会议 (含 ≥2 议程 + ≥2 expert 邀请)

- **前置**:P-1 ~ P-5 全 pass
- **目标**:用 leader 账号 POST `/api/meetings` 创出一个 `mode=auto` 会议,拿到 `meeting_id`,后续 T-02 ~ T-08 引用
- **步骤**(原文复制 — Kimi 必须照搬;Kimi 自己挑 ws 内 1 个 moderator + 2 个 expert agent 的 UUID 填到 `attendee_agent_ids`):

  1. 从 P-5 的 `/tmp/kimi-agents.json` 里挑 1 个 `role==moderator` 的 `id`
     和 2 个 `role==expert` 的 `id`,记为 `MODID`、`EID1`、`EID2`(全 UUID)。
  2. 提交:
     ```
     curl -sS -X POST https://aimeeting.zhzjpt.cn/api/meetings \
       -H "Content-Type: application/json" -b /tmp/kimi-cookie.txt \
       -d '{
         "title":"Kimi Sprint3 WS 验证会",
         "attendee_user_ids":[],
         "attendee_agent_ids":["MODID","EID1","EID2"],
         "agenda":[
           {"title":"议程一:试点方案","time_budget_min":5},
           {"title":"议程二:风险评估","time_budget_min":5}
         ],
         "mode":"auto",
         "description":"Sprint3 端到端 WS 推 验证 — 仅用于自动化,LLM 跑 2 个议程后 done."
       }' > /tmp/kimi-create.json
     cat /tmp/kimi-create.json
     ```
- **Pass 条件**:
  - HTTP 200 / 201
  - JSON 含 `"mode": "auto"`
  - JSON 含 `"id"` 字段(UUID 格式),记为 `MEETING_ID`,后续步骤引用
- **Fail 条件**:
  - JSON 含 `"error"` 字段
  - HTTP 4xx / 5xx
  - `mode` 字段不是 `"auto"`
- **必须复述**:`/tmp/kimi-create.json` 完整原文

### T-02:启动会议 → `status=ongoing`

- **前置**:T-01 pass,有 `MEETING_ID`
- **目标**:把会议从 `scheduled` 切到 `ongoing`,这是 orchestrator 启动的必要条件
- **步骤**:
  ```
  curl -sS -X POST https://aimeeting.zhzjpt.cn/api/m/meetings/<MEETING_ID>/start \
    -b /tmp/kimi-cookie.txt > /tmp/kimi-start.json
  cat /tmp/kimi-start.json
  ```
- **Pass 条件**:
  - HTTP 200
  - JSON 含 `"status": "ongoing"`
  - JSON 含 `"started_at"`(非 null)
- **Fail 条件**:
  - `status` 不是 `"ongoing"`
  - 任何 4xx / 5xx
- **必须复述**:`/tmp/kimi-start.json` 完整原文

### T-03:启动 orchestrator (POST /orchestrate/start)

- **前置**:T-02 pass
- **目标**:trigger auto orchestrator 跑起来。本 Sprint 主测它写完 DB 后会
  broadcast WS 事件
- **步骤**:
  ```
  curl -sS -X POST https://aimeeting.zhzjpt.cn/api/meetings/<MEETING_ID>/orchestrate/start \
    -b /tmp/kimi-cookie.txt > /tmp/kimi-orch-start.json
  cat /tmp/kimi-orch-start.json
  ```
- **Pass 条件**:
  - HTTP 200 / 202
  - JSON 含 `"phase": "running"` 或 `"started": true`(具体字段看后端响应)
  - 不报 `"error"`
- **Fail 条件**:
  - HTTP 400 / 500
  - 响应里 phase 字段是 `"failed"` 或 `"cancelled"`
- **必须复述**:`/tmp/kimi-orch-start.json` 完整原文

### T-04:5 秒后聚合详情显示 phase=running + current_speaker_agent_id 非空

- **前置**:T-03 pass
- **目标**:验证 orchestrator 已开始跑,`auto_state.phase` 已切 `running`。
  当前 sprint 关注的是 _backend 在这一步 broadcast 了 WS 事件_,但 HTTP
  聚合接口同样可验 phase 已落 DB
- **步骤**:`sleep 5 && curl -sS -b /tmp/kimi-cookie.txt https://aimeeting.zhzjpt.cn/api/m/meetings/<MEETING_ID> > /tmp/kimi-detail-t04.json && cat /tmp/kimi-detail-t04.json | jq '{mode, orchestrate_phase, current_speaker_agent_id, orchestrate_turn_count, current_agenda_idx}'`
- **Pass 条件**:
  - HTTP 200
  - `mode == "auto"`
  - `orchestrate_phase == "running"`
  - `current_speaker_agent_id` 是 UUID 或 null(orchestrator 可能在 议程间隙,
    null 也 OK,关键是 phase=running)
  - `current_agenda_idx == 0`(第一个议程)
- **Fail 条件**:
  - `mode` 不是 `"auto"`
  - `orchestrate_phase` 仍为 `"idle"` 或已是 `"failed"` / `"cancelled"`
- **必须复述**:`jq` 输出原文

## 已知不在本次 Sprint 3 范围 (Kimi 见到这些不算 fail)

- **真 chunk-by-chunk streaming**:本次是 blob-style (start + 单 chunk = 完整
  text + end),消息会一次性 blob 进来,不是字符逐个吐字。这是有意的简化,
  Sprint 4 / 5 再上 真 streaming。
- **`agent_recommendation` / `dissent_detected` / `agenda_off_topic` 等 hybrid
  会议 才有的事件**:auto 会议路径里没这些,Kimi 看不到也不算 fail。
- **P95 延迟 数值测量**:Kimi 不能精确测 WS 推 vs 后端写 DB 之间的 ms 级
  延迟,本测验只验 _功能正确性_(消息 / phase 都能到 UI)。延迟数值由 PM
  根据 prod 日志另测。
- **WS 掉线 自动 fallback 到 2.5s 轮询**:这一路要 模拟 WS 掉线 才能测出,
  Kimi 不易模拟(需要 在 Network panel 阻断 WS frame),本次不强测。
  实现层 已加 `conn !== "connected"` 门闸,代码 review 已确认。
- **iOS 微信 WKWebView 特殊行为**:本测验在桌面浏览器跑,不覆盖小程序壳。
- **Sprint 2-3 已 ship 的 1010 行 orchestrator 决策逻辑**(议程切换 / 共识
  收敛 / 议程超时 等):不在 Sprint 3 改动范围,Kimi 不重复测。
