# 🟦 P7 用例 (T-01 ~ T-05)

## T-01 · 出处 链回 chip + focus 跳转 ★ (P7.3)

**前置**: P-3 pass + 找一个 pending Memory 草稿 (浏览器 `/me/profile/sedimentation` Memory tab pending).

**步骤**:
1. 看 草稿 行 (Memory tab pending) 是否 显 chip
2. 若 有 chip, click → 看 跳转 URL

**Pass** (chip 存在 时):
- (a) chip 字面 含 `"📝 来自"` + 数字 + `"句"` + `"→ 看 上下文"`
- (b) chip 是 `<a href="/meeting/<mid>?focus=<ids>">`
- (c) click 跳 转 后 实录页 自动 滚到 那几句 + 高亮

**Pass** (chip 不存在 时 — 老 草稿):
- (d) 行 内 没有 含 "📝 来自" 字样 的 元素 (老 数据 source_line_ids=NULL, 不渲)

**Fail 情景**:
- 草稿 是 新 抽 的 (closure_curator 升级 后) 但 没 chip → 后端 没 写 source_line_ids
- chip 显 但 click 后 没 滚到 高亮 → focus 路径 出错

**复述**: 截 草稿 行 截图 + 若 有 chip click 后 实录页 截图.

## T-02 · 草稿 inline 编辑 + 一键 通过 ★ (P7.1)

**前置**: T-01 pass + 选 一个 pending 草稿 → 点 "查看 / 审批 →" 进 dialog.

**步骤**:
1. 看 dialog 中 "拟写入内容" 区, 右上角 应有 `"✏️ 编辑"` 链 (`data-testid="memdraft-edit"`)
2. 点 编辑 → 应 看到 textarea (替代 老 `<pre>`) + 重要度 slider
3. 改 内容 (例 末尾 加 一句 "(已审改)") + 调 slider
4. 点 `"✓ 保存 并 通过"` (`data-testid="memdraft-save-and-approve"`)
5. dialog 关闭, 列表 应 自动 刷新 — 该 草稿 移到 已批准 tab

**Pass**:
- (a) ✏️ 编辑 按钮 存在 + click 后 textarea 出现
- (b) textarea 显 老 内容 可 编辑
- (c) 重要度 slider 存在 + 显 数值 (0.00 ~ 1.00)
- (d) "保存 并 通过" 后 toast 含 `"✅ 已 保存 编辑 + 批准 入库"`
- (e) 切 到 "已批准" tab 看 该 草稿: content 显 改后 内容 + importance 显 改后值

**Fail 情景**:
- 没 编辑 按钮 → 前端 没 deploy
- 保存 失败 (HTTP 4xx/5xx) → 看 backend logs

**复述**: 截 编辑 前 / 编辑 后 / 已批 列表 三 张 截图.

## T-04 · 拒绝 二选一 (弃用 vs 退回 LLM) ★ (P7.4)

**前置**: T-02 pass + 仍 有 ≥ 1 个 pending Memory 草稿 → 进 dialog → 点 "驳回…"

**步骤** (两 子用例):

### T-04a · 弃用 (默认)
1. 应 看到 两 个 radio 选项: `🗑 弃用` (默认 选中) / `↩ 退回 LLM 重抽`
2. 留 默认 → textarea placeholder 应 是 `"例: 这条事实 有歧义"`
3. 写 一句 注 (or 不写) → 点 `"🗑 确认 弃用"`

**Pass**:
- (a) 两 个 radio 字面 一致
- (b) 默认 选中 = 弃用
- (c) "🗑 确认 弃用" 按钮 红色
- (d) toast 含 `"已 弃用"` 字面
- (e) 已驳回 列表 该 草稿 行 显 rose 短chip 含 `"🗑 弃用"`

### T-04b · 退回 LLM (feedback)
1. 同 dialog 切 radio 到 `"↩ 退回 LLM 重抽 — 写 反馈 (≥ 5 字)"`
2. textarea placeholder 变 `"例: 主语 不对, 张三 不是 反对方..."`
3. **不写** 反馈 → 点 确认 → 应 toast 报错 `"退回 LLM 时 反馈 必填 ≥ 5 字"`
4. 写 ≥ 5 字 → 确认

**Pass**:
- (a) 切 radio 后 placeholder 变
- (b) 空 反馈 拒绝 + toast 报错
- (c) 满足 ≥ 5 字 后 toast 含 `"已 退回 LLM"`
- (d) 已驳回 列表 行 显 amber chip 含 `"↩ 退回 LLM · <反馈>"`

**Fail 情景**:
- radio 不显 → 前端 没 deploy
- 不验证 ≥ 5 字 → 后端 / 前端 校验 缺失

**复述**: 两 子用例 各 截图 + 已驳回 行 chip 字面.

## T-05 · 审批 通知 ★ (P7.5)

**前置**:
- T-02 / T-04a / T-04b 已 触发 至少 一次 通过 / 弃用 / 退回. **关键: 草稿 来源 (会议 / 任务 创建人) ≠ 当前 owner 账号** — 否则 自决 自通知 会被 skip.
- 若 owner 是 唯一 source 创建人, 此用例 SKIP.

**步骤**:
1. owner 操作 完 → 切 到 草稿 来源 用户 账号 (例 leader `demo.lijg`)
2. 看 顶部 chrome NotificationBell (🔔) — 应 显 unread badge (+1 or +N)
3. 点开 Bell → 看 列表

**Pass**:
- (a) bell 角标 +1 (or +N 取决于 触发 几次)
- (b) 通过 触发: 一行 含 `"✅ <审批人> 批准了你提的长期记忆"` 字面
- (c) 弃用 触发: 一行 含 `"⛔ <审批人> 弃用了你提的长期记忆"` 字面
- (d) 退回 触发: 一行 含 `"↩ <审批人> 退回 你提的长期记忆 — <feedback>"` 字面 (yellow severity)
- (e) 点 任一 行 → 跳 `/me/profile/sedimentation`

**Fail 情景**:
- bell 没 +1 → 通知 没 emit; 看 backend logs 看 是否 _notify_decision 被 skip (target=自己)
- 字面 不一致 → frontend renderer 改错

**复述**: 截 bell 截图 + 列出 字面.


**Pass**:
- (a) HTTP 200 + JSON `{"ok":true,...}`
- (b) 渲 `data-testid="moderator-banner-decision_summary"` 紫色 banner
- (c) 含 字面 `"🧭"` + `"主持人 即将 帮 收口"` + `"测试 议程项"`
