## 顶部 6 条死规矩 (反幻觉)

1. 只看真实页面 / 真实 HTTP 响应.
2. 每步判定必须基于本步实际看到的数据.
3. 复述要原文 (带引号的精确字符串).
4. 不允许跳步. 前一步 fail → 后续 BLOCKED.
5. 不允许编造账号.
6. 每用例 4 段必须给齐: 实际看到 / 判定 / 失败理由 / 证据.

## 预检 P-1 ~ P-3

### P-1 · 登录 owner, 拉 /m/tasks 看到 cta
- 登录 `/login`, 跳 /m
- 进 `/m/tasks`
- segment 默认 "等你处理"
- **Pass**: 至少看到一张卡, 卡 footer 有两个按钮 (主 + 副).

### P-2 · 后端 mobile tasks endpoint 给出新 cta 文案
```
curl -s -b cookies https://aimeeting.zhzjpt.cn/api/m/tasks
```
- 找任一 `kind == "confirm"` 的 item
- **Pass**: 该 item `cta_primary == "确认"`, `cta_secondary == "驳回"` (不是 "确认 派发" / "改 一下").
- **Fail**: 旧版文案还在 → 后端没上线.

### P-3 · 任一 action / draft / agenda-advance endpoint 401 (未登录)
```
curl -i -X PATCH https://aimeeting.zhzjpt.cn/api/meetings/00000000-0000-0000-0000-000000000000/actions/00000000-0000-0000-0000-000000000000 -H 'content-type: application/json' -d '{"status":"done"}'
curl -i -X POST https://aimeeting.zhzjpt.cn/api/memory-drafts/00000000-0000-0000-0000-000000000000/approve
curl -i -X POST https://aimeeting.zhzjpt.cn/api/meetings/00000000-0000-0000-0000-000000000000/agenda-advance
```
- **Pass**: 三个都返 401. **Fail**: 任一 5xx / 200.

## T-01 · ActionItem 确认 (primary CTA) — 卡消失 + Toast

### 前置
P-* PASS. /m/tasks segment "等你处理" 看到至少一张 `kind=confirm` 卡.

### 步骤
1. 记下任一 confirm 卡的 source_meeting_id 和 id (用 devtools 抓 API JSON).
2. 在网页上点该卡的 **主按钮 "确认"**.
3. 等待 (按钮应短暂变 "处理中…" disabled).
4. 观察:
   - 是否弹出 toast?
   - 卡是否从 "等你处理" 列表消失?
   - 切到 "已完成" tab — 该卡是否出现在那?

### Pass (全部)
- 主按钮在请求中文字 "处理中…" + disabled (opacity 降低), 请求完恢复.
- 屏幕底部出现绿色 toast "✓ 已确认" (字符串可有 "✓ "前缀, 主体为 "已确认").
- 该卡从 "等你处理" tab 消失.
- 切到 "已完成" tab, 该卡 (相同 id) 出现, kind=done, 状态 chip "已完成".
- 后端确认: `curl GET /api/m/tasks` 该 id 的 group 已变成 "done".

### Fail
- 弹了 alert 而不是 toast.
- 卡没动 (列表没变).
- Toast 是红色错误而不是绿色 success.

### 必须复述
- 点击前后两次 `curl GET /api/m/tasks | jq '.items[] | select(.id=="<id>")'` 完整 JSON.
- toast 截图.
- 卡消失前后截图.

## T-02 · ActionItem 驳回 (secondary CTA)

### 前置 P-* PASS.

### 步骤
- 在 "等你处理" 找另一张 confirm 卡 (避免跟 T-01 同一条).
- 点 **副按钮 "驳回"**.

### Pass
- toast "✓ 已驳回".
- 卡从 "等你处理" 消失.

## T-05 · 议程推进 (StickyActionBar)

### 前置
- 至少 1 个 ongoing meeting 含 agenda + 我有 controller 权限.
- 进入该 meeting `/m/meetings/<id>`.

### 步骤
1. 看 sticky bar 底部 — 应有 "推进议程 →" 按钮.
2. 记当前 `current_agenda_idx` (用 devtools 抓 `/api/m/meetings/<id>` 的 JSON).
3. 点按钮.
4. 观察按钮文案变 "推进中…" disabled, 几百 ms 后:
   - 弹绿色 toast "✓ 议程已推进".
   - StageChipsRow 顶部 active 项 chip 向右移一项.
   - 当前议题主卡内容变 (title 跟着 agenda 下一项 + insights 清).

### Pass (全部)
- toast 绿色 "✓ 议程已推进".
- 后端: 再 GET `/api/m/meetings/<id>`, `current_agenda_idx` 比之前 +1 (或变 null + `is_agenda_complete=true`).
- 主区域 CurrentTopicCard 标题切到 agenda[next_idx].title.

### Fail
- toast 红色错误.
- 按钮点了无反应 (无变化, 无 toast).
- 后端 `current_agenda_idx` 没变.

### Skip
- 没 ongoing meeting / 没 agenda / 我没 controller 权限 (sticky 显示 "召 AI" 而不是 "推进议程") — 标 SKIP.

### 必须复述
- 点击前 + 后两次 GET `/api/m/meetings/<id>` 的 `current_agenda_idx` + `is_agenda_complete` + `current_topic_title` 三字段.
- toast 截图.

## 反幻觉自检

- "卡消失" 必须前后两张截图佐证.
- "toast" 必须截图含字面 "✓" 或 "⚠" 前缀.
- 后端字段变化必须前后两次 curl 输出, 不能只一次.
- 不允许"应该"/"大概". skip 必须给出原因 (eg "我看了 5 张卡, 全是 kind=tracking, 没 confirm").
