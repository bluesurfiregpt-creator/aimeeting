## T-01 · 专家工卡: 用「最近会议 + 任务汇总」替换「最近 AI 智囊产出」

### 前置
P-1 / P-2 / P-3 全 PASS.

### 目标
确认列表页工卡里, 旧的 "TypeChip + 一行 AI 产出" 区段消失, 改成 「最近会议」「任务」两个小卡片.

### 步骤
1. 进 `https://aimeeting.zhzjpt.cn/m`.
2. 顶部 segment 切到「专家视角」.
3. 找到第一张工卡 (任何 active expert), 仔细看下半区.
4. 同时手动调 `GET /api/m/agents/workboard` (Cookie 同会话), 看 JSON.

### Pass 条件 (全部满足)
- 页面: 工卡下半区出现 **两个并排的小卡**, 一个标题 "最近会议", 一个标题 "任务".
- 「最近会议」小卡: 至少看到 1 条 `MM/DD + 会议标题` 行 (如果该 agent 真有参会). 若 `recent_meetings == []` 则文本为 "未参会".
- 「任务」小卡: 看到 `数字 + 进行中 / 已完成 / 超期` 任一行; 若 `tasks.total == 0` 文本为 "未分配".
- API JSON 里每个 `agents[*]` 含字段: `recent_meetings`, `tasks` (`{total, open_count, done_count, overdue_count}`), **不含** `recent_insights`, **不含** `meetings_count`, **不含** `insights_count`.
- DOM 上找不到 TypeChip (旧版那种 "建议/风险/洞察" 五色 chip).

### Fail 条件
- 工卡内仍出现 TypeChip 文字 (建议/风险/洞察/思路/决策建议).
- API JSON 含 `recent_insights` 字段.
- 「最近会议」「任务」其中一卡完全不存在 (而不是显示 "未参会"/"未分配").

### 必须复述
- 完整粘贴 `agents[0]` JSON 对象 (不省略 任何字段).
- 截图整张第一张工卡 (能看到 header + 下半两个小卡片).

## T-03 · 会议列表 row 显示「真人数 + AI 数」

### 前置 P-* PASS.

### 步骤
1. 进 `https://aimeeting.zhzjpt.cn/m/meetings`.
2. segment 默认在「进行中」(若为空切「已结束」直到看到列表).
3. 看任一会议卡 footer 那行 chip.
4. 调 `GET /api/m/meetings` 看 JSON.

### Pass 条件
- footer 出现 `👤 数字` (真人参会) 以及 / 或 `🤖 数字` (AI 参会) chip — 至少 1 个.
- API JSON `items[*]` 每条含字段 `users_count`, `agents_count`. (允许值为 `0`, 但字段必须存在.)
- 如果某会议 `agents_count == 0`, 该 chip 不展示; 这是正确的, 不算 fail.

### Fail
- 看不到 `👤` 或 `🤖` 任意一个 chip, 且至少有一条会议 API 返回 `users_count > 0` / `agents_count > 0`.
- API JSON 缺 `users_count` / `agents_count` 字段.

### 必须复述
- 完整粘贴 `items[0]` JSON.
- 截图列表内任 1 会议卡 (能看清 footer chips).

## T-04 · 会议列表 row 显示「计划时长」

### 前置 P-* PASS.

### 步骤
1. 仍在 `/m/meetings`.
2. 找一个 status=ongoing 的会议 (若没有, 看 status=finished).
3. 看顶部 header 上紧跟状态 chip 后的副信息文字.
4. API JSON 对照查 `planned_minutes` 字段.

### Pass 条件
- 若 `planned_minutes != null` 且 status=ongoing: 文本含字面值 `计划` + 数字 + `min`. e.g. `已 12 / 计划 45 min`.
- 若 status=finished + `planned_minutes != null`: 文本含 `用时 X / 计划 Y` 形式.
- 若 status=scheduled + `planned_minutes != null`: 文本含 `计划 N min`.
- 若 `planned_minutes == null`: 文本不出现 "计划" 二字 — 是 OK 的, 不算 fail.
- API JSON `items[*]` 含字段 `planned_minutes` (允许为 null).

### Fail
- 一个 `planned_minutes != null` 的会议在 UI 里完全找不到 "计划" 二字.
- API JSON 缺 `planned_minutes` 字段.

### 必须复述
- 找到的会议的 `items[i]` 完整 JSON.
- 截图该会议 row.

## T-05 · ongoing 超时变色 (overshoot)

### 前置 T-04 PASS.

### 步骤
- 必须有一个 status=ongoing 且 `minutes_total > planned_minutes` 的会议. 若没有可暂时 skip.
- 看该会议状态 chip 后的时长文案颜色.

### Pass
- 文案 `已 X / 计划 Y min` 以 **琥珀色 (text-amber-300/90)** 渲染, 视觉上偏黄/橙, 跟其他 row 的灰色副信息不同.

### Fail
- 文案为灰色, 跟其他 row 一样, 没有视觉超时提示.
- 文案完全消失.

### 必须复述
- 截图该会议 row (能看到副信息颜色).
- DOM 类名包含 `text-amber-`. (用 devtools 看, 截图 inspector.)

### Skip 允许
若整个列表里没有 actual > planned 的 ongoing 会议, 标 SKIP 并写明原因. **不**算 fail.
