1. 只看真实页面 / 真实 HTTP 响应.
2. 每步判定必须基于本步实际看到的数据.
3. 复述要原文 (带引号的精确字符串).
4. 不允许跳步. 前一步 fail → 后续 BLOCKED.
5. 不允许编造账号.
6. 每用例 4 段必须给齐: 实际看到 / 判定 / 失败理由 / 证据.

## 预检 P-1 ~ P-2

### P-1 · 登录 owner + 看到至少 1 条 draft 等审
- 登录 owner.
- 进 /m/tasks → "等你处理".
- 找一张 `kind=approve_draft` 卡 (蓝紫色 "待审" chip, 副按钮文案 "驳回").
- **Pass**: 至少 1 张可见.
- **Fail / Skip**: 没 draft 时, 全部 T-* SKIP.

### P-2 · 进 /m/insights → "待我审" tab 也看到同张草稿
- 切到 /m/insights → "待我审".
- 同一张 draft 应在这里也出现 (两个入口共享同一份数据).

## T-01 · /m/tasks 草稿驳回 → 弹 sheet (不直接 cancel)

### 前置 P-* PASS.

### 步骤
1. 在 /m/tasks "等你处理" tab, 找一张 `kind=approve_draft` 卡, 点 **副按钮 "驳回"**.
2. 观察:
   - 屏幕底部弹起 sheet (跟 P4.2 召 AI sheet 同风格).
   - sheet 顶部把手 + 标题: "驳回 — 可选告诉 AI 哪不对"
   - 副标题: "写了 AI 下次会更准. 不写也行, 直接丢弃."
   - 一个引用块显示 draft 内容 (前 ~80 字 line-clamp-3).
   - textarea (rows=4), placeholder 含 "例: 这条不该入个人库..."
   - 右下角字数计数 "0/500".
   - 底部两按钮: 灰边 "仅驳回" + 蓝主 "驳回并反馈".

### Pass (全部)
- sheet 弹出.
- 蓝主按钮 "驳回并反馈" disabled (因为没文字).
- 灰按钮 "仅驳回" enabled.
- 草稿 card 在 sheet 弹起后**仍在列表里** — 后端**还没**被请求 (Network 看无 reject 请求发出).

### Fail
- 点 "驳回" 后无 sheet, 直接发 reject 请求 (旧 P4 行为).
- sheet 显示但状态对不上 (按钮 enabled 不对).
- 后端被请求 (Network 有 reject API 出去).

### 必须复述
- 整个 sheet 截图.
- Network panel 截图含 (无 reject 请求).

## T-02 · 点 "仅驳回" (空文本) → discard 行为

### 前置 T-01 sheet 已打开.

### 步骤
1. 不输入任何文字.
2. 点 **灰按钮 "仅驳回"**.
3. 观察:
   - 灰按钮变 "处理中…" disabled.
   - 几百 ms 后 sheet 收起.
   - 绿色 toast "已驳回".
   - 该 draft 从列表消失.
4. devtools Network 看实际发送 body: 应为 `{"kind":"discard"}`.

### Pass (全部)
- toast 绿色 "已驳回".
- Network 请求 body 是 `{"kind":"discard"}` (不带 feedback_text).
- 列表中该 draft 消失.
- 后端: 该 draft.status → "rejected".

### Fail
- toast 红色错误.
- 请求 body 含 feedback_text 字段 (说明传错).

### 必须复述
- Network XHR 截图 (含 request body + response).
- toast 截图.

## T-03 · 重打开 sheet → 输入文本 → 主按钮变 enabled

### 前置 P-* PASS, 至少还有 1 张 draft.

### 步骤
1. 在 /m/tasks 找另一张 draft, 点 "驳回" 弹 sheet.
2. 在 textarea 输入: "这条重复了已有记忆, 重新抽" (字面值, 18 字).
3. 观察:
   - 字数计数变 "18/500".
   - 蓝主按钮 "驳回并反馈" 从 disabled 变 enabled (颜色饱满).
   - 灰按钮 "仅驳回" 仍 enabled.

### Pass
- 字数 = 18.
- 蓝主按钮可点.

### Fail
- 字数计数错.
- 按钮状态没变.

## T-04 · 点 "驳回并反馈" → feedback 行为

### 前置 T-03 已输入文本.

### 步骤
1. 点蓝主按钮 "驳回并反馈".
2. 蓝按钮变 "处理中…" disabled.
3. sheet 收起.
4. toast 绿色 "已驳回并反馈".
5. 该 draft 从列表消失.
6. devtools Network 看 body: 应为 `{"kind":"feedback","feedback_text":"这条重复了已有记忆, 重新抽"}`.

### Pass (全部)
- toast "已驳回并反馈".
- Network body 含 `kind:"feedback"` 和 `feedback_text` 字段, 文案字面一致.
- 列表中该 draft 消失.

### Fail
- toast 跟 T-02 一样 "已驳回" (没区分 feedback / discard).
- 请求 body 缺 feedback_text.

### 必须复述
- Network 请求 body 原文.
- toast 截图.

## T-05 · /m/insights "待我审" tab 同样路径

### 前置 P-* PASS, 仍有 draft.

### 步骤
1. 进 /m/insights segment "待我审".
2. 找一张 draft, 点副按钮 "驳回".
3. 应弹同款 sheet.
4. 测一遍: 不输入 → "仅驳回" → toast → 卡消失 → Tab 3 "已入库" 不增加 (驳回不入库).

### Pass
- /m/insights "待我审" tab 上的驳回行为跟 /m/tasks 一致 (弹 sheet, 按钮一致, body 一致).
- Tab 3 已入库 count 不变.

### Skip
- 无 draft, SKIP.

## T-06 · 取消 sheet → 草稿不动

### 前置 P-* PASS, 还有 draft.

### 步骤
1. 找一张 draft, 点 "驳回" 弹 sheet.
2. 点 sheet 上方背景区 (黑色遮罩) — 或浏览器后退键.
3. 观察:
   - sheet 收起.
   - 草稿 card 仍在列表里 (没被驳).

### Pass
- 草稿没消失.
- 后端无 reject 请求 (Network 看).

### Fail
- 草稿被误驳.

## T-07 · action item 的"驳回"不弹 sheet (维持现状)

### 前置 P-* PASS, /m/tasks 有 kind=confirm 卡.

### 步骤
1. 在 /m/tasks "等你处理" tab, 找一张 `kind=confirm` (action item) 卡.
2. 点副按钮 "驳回".

### Pass
- **不弹** sheet — 直接发 PATCH 请求 cancel 该 action item.
- 卡消失, toast "已驳回" (老行为).

### Fail
- action item 也弹了 sheet (说明逻辑误判).

### 必须复述
- 点击前 + 后两次 GET /api/m/tasks (action item 应该没了).
