## 3. 测试用例 T-01 ~ T-07

### T-01 · 准备:创建会议 + 跑出分歧

**前置**:P 全 pass
**目标**:为后续用例准备一个 已经跑完且至少有 1 处分歧的 auto 会议

**步骤**:
1. 创建 auto 会议:
   - 标题:`kimi-v26.3-07-T01-<UNIX 时间戳>` (字面例 `kimi-v26.3-07-T01-20260513120000`)
   - 议程 **3 项**,标题刻意让 AI 产生互斥立场,推荐:
     - 议程 1:`新增住宅项目优先采用预制混凝土还是现浇`
     - 议程 2:`小区物业费上调与否`
     - 议程 3:`保障房分配优先按户籍年限还是按申请时间`
   - 勾 **4 个 expert** (任意 4 个 bound 的 expert)
   - 提交
2. 跳到 `/meeting/<id>/orchestrate` 后,点 🚀 启动
3. 等到 phase 变 `done` (1~3 分钟,看 LLM 速度)
4. 看 顶部 banner

**Pass 条件**:
- phase 字面变 `"done"`
- 顶部 banner 出现 `⚠️ 本场会议 有 N 处分歧 待你裁决`,N **≥ 1**
- banner 上能看到 "打开裁决面板" 按钮(数据 testid=`open-review-modal`)
- API 验证:
  ```
  GET /api/meetings/<id>/consensus
  ```
  - 返回数组长度 字面 等于 `3`
  - 至少一个元素 `needs_human_review: true` 且 `dissents` 数组 非空
  - 每个 dissents 元素 包含 `point` `summary` `involved_agents` 三字段

**Fail 条件**:
- 3 议程都 0 分歧 → fail("LLM 没识别到互斥;重试或换更对立议程")
- banner 没出现 → fail("v26.3-07e 前端 banner 失效")
- needs_human_review 全 false → fail("orchestrator 端写库异常")

**必须复述**:
- 完整 `GET /consensus` 数组(每个元素的 needs_human_review + dissents 数组前 100 字)
- 顶部 banner 截图
- 记下 meeting_id 给后续用 `<M1>`,至少一个 needs_human_review=true 的 agenda_idx `<A1>`,该议程的 dissent 数 `<D1>`

### T-02 · 打开裁决 modal + 校验字段渲染

**前置**:T-01 pass
**步骤**:
1. 点 "⚖️ 打开裁决面板" 按钮
2. 看弹出来的 modal (`data-testid="dissent-modal"`)
3. 对每个待裁决议程卡片:
   - 顶部应有 议程编号 + 标题 + 分歧数
   - 每个 dissent 应有 4 个 radio (字面文字:`采纳 A 方`、`采纳 B 方`、`折衷裁决`、`搁置 / 待后续`)
   - 应有 textarea 占位符含 `≥10 字`
   - 应有 "✓ 裁决并归档" 按钮 (testid=`submit-<agenda_idx>`)

**Pass 条件**(全部):
- modal 显示,可关闭
- pending 议程数 字面等于 T-01 拿到的 needs_human_review=true 议程数
- 每个 dissent 卡片 4 个 radio 全渲染
- textarea 初始为空
- rationale 字符计数显示 `0/10+ 字`

**Fail 条件**:
- modal 点不开 → fail
- radio 只有 3 个或 5 个 → fail
- 4 个 radio 文字 与上面四个字面值任一不匹配 → fail

**必须复述**:
- 4 个 radio 的精确文字
- modal 顶部标题 原文
- 截图 modal 全屏

### T-03 · 校验提交前的本地校验 (rationale < 10 字)

**前置**:T-02 pass,modal 打开
**步骤**:
1. 议程 `<A1>` 的所有 dissent,radio 都选 `采纳 A 方`
2. rationale 每个填 `abc` (3 字)
3. 点 "✓ 裁决并归档"

**Pass 条件**:
- **没有** 发出 `POST /consensus/<A1>/review` 请求(网络面板)

**Fail 条件**:
- 请求发出了 → fail("前端校验失效,后端兜底了")
- 没有任何错误提示 → fail("校验静默失效")

**必须复述**:错误提示原文 + 截图

### T-04 · 提交合法裁决 → 落库 + 沉淀

**前置**:T-03 pass
**步骤**:
1. 议程 `<A1>` 所有 dissent:
   - 第 1 个 radio 选 `采纳 A 方`,rationale 填 `我决定优先采纳A方专家的立场理由如下扩充至15字以上`(≥10 字,且写在一行)
   - 第 2 个 (若有) radio 选 `折衷裁决`,rationale `综合双方观点取折衷做长期跟踪不立即定论`
   - 第 N 个 (若有) radio 选 `搁置 / 待后续`,rationale `下次会议带回深入讨论暂不决议先调研`
2. 点 "✓ 裁决并归档"
3. 等 5 秒
4. API 验证:
   ```
   GET /api/meetings/<M1>/consensus
   ```

**Pass 条件**(全部):
- `POST /api/meetings/<M1>/consensus/<A1>/review` 返回 status 字面等于 `200`
- 修改后议程 `<A1>` 的 `reviewed_at` 字段 字面 非 null,是 ISO 字符串
- `reviewed_by_user_id` 字段 字面 是 UUID(非 null)
- `review_decision` 字段 字面 是 JSON 字符串数组,解析后 长度 等于 dissent 数
- 数组每元素含 `dissent_idx` `action` `rationale` 三字段,action ∈ {`"pick_a"`, `"compromise"`, `"defer"`}
- toast 出现 `裁决完成` 字串
- modal 里 该议程消失 / 移到 "已裁决" 区

**Fail 条件**:
- POST 非 200 → fail,粘 response body
- reviewed_at 还是 null → fail
- review_decision 解析失败 → fail
- modal 没刷新 → fail("onReviewed 回调失效")

**必须复述**:
- `POST /review` 的完整 response JSON(整个对象)
- 解析 review_decision JSON 后的内容
- meeting_id, agenda_idx,reviewed_at 时间戳

**必须复述**:response 完整 body

### T-06 · 沉淀异步生效:KB 出现 [召集人裁决] 文档

**前置**:T-04 pass,等 **15 秒** (LLM 4 段 + embedding 累计)
**目标**:验证 schedule_consensus_consolidate 真的跑了

> 🚨 **重要**:不要 去会议详情页找"知识库 Tab" — 这个 Tab 不存在,本期产品设计就是不在会议详情页放.
> 沉淀的 KB doc 验证 必须 走 KB API,不依赖 任何 UI tab.

**步骤(纯 API,不点 UI)**:

1. **找到涉及的 expert agent ID**.从 T-01 复述的 consensus JSON 里,议程 `<A1>` 的 `dissents[0].involved_agents` 数组,取第一个名字记 `<AG_NAME>`(例 `"国家政策专家(粤深)"`).

2. **API 查 该 agent 的 knowledge_base_ids**:
   ```
   GET /api/agents
   ```
   - 返回数组里找 `name == <AG_NAME>` 的元素,记其 `knowledge_base_ids[0]` 为 `<KB_ID>`
   - 如果 `knowledge_base_ids` 为空,先去 `GET /api/knowledge-bases` 列所有 KB,找 name 含 `<AG_NAME>` 的(应是 `"<AG_NAME> · 任务沉淀"`),记 id 为 `<KB_ID>`

3. **API 查该 KB 下的文档**:
   ```
   GET /api/knowledge-bases/<KB_ID>/documents
   ```

**Pass 条件**(全部):
- 列表里出现 至少 1 条 `filename` 以字面字符串 `[召集人裁决]` 开头的 doc
- 该 doc `source_type` 字面等于 `"consensus_dissent"`
- 该 doc `source_agent_id` 字段 非 null
- 该 doc `status` 字面等于 `"ready"`(若是 `"embedding"`,再等 20 秒重查;`"failed"` → fail)
- 该 doc `chunk_count` 字段 ≥ 1

**Fail 条件**:
- 15+15+10 秒都没出现 → fail("沉淀 worker 没跑,看后端 log `consensus_consolidator`")
- source_type 是 `"manual"` 或 `"task"` → fail("source_type 写错")
- status `"failed"` → fail,粘 error_message 字串

**必须复述**:
- doc 的完整 JSON(id / filename / source_type / source_agent_id / chunk_count / status)
- 找到的 KB id
