# Kimi 测试用例 · v26.5-P0 综合 — 角色重设计 (后端 ABAC + UI 行为)

> **场景**:综合验证 v26.5-P0 角色重设计 — 5 角色矩阵(owner/admin/leader/manager/member)+ 后端 ABAC 收口 + 前端 role-aware UI 全套
> **commit**:`9b87209`(后端 P0 `ab300c3` + 前端 UI P0 `9b87209`)
> **部署**:`https://aimeeting.zhzjpt.cn`
> **测试对象**:Kimi
>
> 🚨 **环境前提**:
> - 已配 `PLATFORM_ADMIN_EMAILS=bluesurfiregpt@gmail.com`
> - DB 干净:1 默认 workspace + 1 超管 user + 21 agent + 20 KB
>
> **目标验证 6 件事**:
> 1. `manager` 角色枚举(邀请下拉可选 + me.role 字面 = `"manager"`)
> 2. **顶栏入口** role-aware:owner/admin/leader/manager 看到 ⚙️ + 📊;member 看不到
> 3. **`/admin/*` Tab 渲染**:leader+ 看 10 tab;manager 仅 3 tab;member 跳 /me
> 4. **后端写权限 ABAC**:KB / agent / memory / dashboard scope 收口
> 5. **agents 页 UI**:manager 看到 别人 agent 上有 🔒;改 primary_user_id 字段灰禁
> 6. **错误提示分类**:全部 🟡 黄 warn `[权限不足]`,标题"权限不足",不是"请求失败 (403)"

---

## 0. 给 Kimi 的 6 条死规矩

1. 只看真实页面 / HTTP 响应,推测一律 fail
2. 每步判定基于本步实际数据,不复用上一步
3. 复述原文带引号(`role: "manager"` / `scope_label: "我管理的 1 个 AI 专家"`)
4. 顺序依赖.前置 fail → 后续 BLOCKED
5. 不允许编账号 / URL / 字段名
6. 4 段:实际看到 / 判定 / 失败理由 / 证据

**额外**:本用例 需要 你 自己创建测试客户 workspace + 邀请 3 个角色(owner/manager/member)的 user.记好凭证以备后用.UI 验证 必须真截图,不允许"推测看到".
