## 测试用例

### A. KB.owner_agent_id (v26.5-02a)

#### T-01 · owner 创建 KB 时 指定 owner_agent = AGENT_1
- 步骤: OWNER 进 `/admin/knowledge` → 新建 KB "T01 KB" → 选 owner_agent_id = AGENT_1 → 创建
- Pass: KB 列表显示 🤖 AGENT_1 徽章, KB.owner_agent_id 返回 AGENT_1.id
- 证据: 截图 + GET /api/knowledge-bases body

#### T-02 · MGR1 (AGENT_1.primary) 给 "T01 KB" 上传文档 → 200
- 步骤: MGR1 浏览器 进 `/admin/knowledge/{T01_KB_ID}` → 上传任意 PDF
- Pass: 上传成功, 列表显示文档. POST `/api/knowledge-bases/{id}/documents` 200
- Fail: 403 → 说明 ABAC 没生效

#### T-03 · MGR2 给 "T01 KB" 上传文档 → 403 ★
- 步骤: MGR2 浏览器 进同一 KB 页 → 上传
- Pass: POST 403 `[权限不足] 写此 KB 需要 owner/admin/leader, 或该 KB 归属 AI 的 primary_user (manager)`

#### T-04 · MGR1 进 /admin/knowledge → 看到 ⭐ 我维护
- Pass: T01 KB 卡片有 amber `⭐ 我维护`, 别的 KB (无 owner) 显示 🔒
- 编辑按钮: T01 ✏️ 可点, 别的 🔒

#### T-05 · MGR1 编辑 T01 KB 改 description → 200
- 步骤: 点 ✏️ → 改简介 → 保存
- Pass: PATCH `/api/knowledge-bases/{id}` 200, 列表显示新简介

#### T-06 · MGR1 改 T01 KB.owner_agent_id → 403 ★
- 步骤: 编辑 dialog 中 ↑ owner_agent select 应该 是 纯文字 (manager 不可改)
- 注: 这是 UI 层防御, 后端也拒. 如果尝试 curl PATCH owner_agent_id, 后端 应 403 `仅 owner / admin / leader 可指派 / 转移`

#### T-07 · 删 owner_agent 后 KB 退到 admin-only
- 步骤: OWNER 删除 AGENT_1 → 看 T01 KB
- Pass: T01 KB.owner_agent_id 变 NULL (ON DELETE SET NULL), MGR1 失去写权
- 跳过 T-07 如不想真删 AGENT_1, 可手 SQL 验证

### B. LongTermMemory.agent_id (v26.5-02b)

#### T-08 · MGR1 写一条 memory 归属 AGENT_1 → 200
- 步骤: MGR1 进 `/admin/memory` → 选 agent_id = AGENT_1 → 写内容 → 添加
- Pass: POST `/api/memory` 200, 列表显示带 `🤖 AGENT_1` 徽章

#### T-09 · MGR1 写 workspace 通用记忆 (agent_id 留空) → 表单不允许 + 后端 403
- 步骤: MGR1 form 试 不选 agent_id → 表单按钮提示 "manager 必须指定归属 AI"
- (后端 path: 如果绕过前端 curl 不带 agent_id, 后端 require_leader_or_admin → 403)

#### T-10 · MGR2 写 memory 归属 AGENT_1 → 403 ★
- 步骤: MGR2 form 选 AGENT_1, 但前端 select 应该 不显示 AGENT_1 (只显示 MGR2 自己 primary 的)
- 如果 curl 强行 POST agent_id=AGENT_1 → 后端 403 `[权限不足] 写此 AI 的记忆 需要 ...`

#### T-11 · MGR1 删自己的 memory → 200
- Pass: DELETE 200

#### T-12 · MGR2 删 MGR1 的 memory → 403 ★
- Pass: DELETE 403

### C. 沉淀审批 gate (v26.5-02c)

预备: 创建一个任务, OWNER 派给 AGENT_1, 让 MGR2 (非 AGENT_1.primary) 办结.

#### T-13 · MGR2 办结 (curator≠primary) → 创建 SedimentationDraft, 不立即写 KB
- 步骤: MGR2 进 `/me` → 找 这个 task → submit → OWNER 审核 → approve → task done
- Pass: 
  - 立即检查 GET `/api/sedimentation-drafts` (MGR1 视角) → 1 条 status=pending
  - GET `/api/knowledge-bases/{T01_KB_ID}/documents` → 没新加 (KB 没写入)
  - MGR1 顶栏 / `/me/profile` 看到 🔔 1 个 待审批

#### T-14 · MGR1 进 /me/profile 看待审批 + 批准 → KB 真的写入
- 步骤: MGR1 进 `/me/profile` → 🔔 section 看到 task → 点查看/审批 → 看 拟摘要 → 点 ✅ 批准沉淀
- Pass:
  - POST `/api/sedimentation-drafts/{id}/approve` 200
  - draft.status = approved
  - GET `/api/knowledge-bases/{T01_KB_ID}/documents` → 新增 1 个 status=ready, source_type=task
  - MGR2 收到通知 `kb_sedimentation_approved`

#### T-15 · MGR1 驳回另一个 draft → KB 不变
- 步骤: 再制造一个 draft (走 T-13 重复), MGR1 选驳回 + 填理由
- Pass:
  - POST `/api/sedimentation-drafts/{id}/reject` body `{reason}` 200
  - draft.status = rejected, decision_reason 保存
  - KB 没增加文档
  - MGR2 收 通知 `kb_sedimentation_rejected`

#### T-16 · MGR2 自己办结 自己 primary 的 agent task → 直接沉淀 (不走 gate)
- 步骤: OWNER 派 task 给 AGENT_2, MGR2 (= AGENT_2.primary) 办结
- Pass: task done → 直接写 KB (无 SedimentationDraft 创建). GET /api/sedimentation-drafts MGR2 视角 0 条 pending.
