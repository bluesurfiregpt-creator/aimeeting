## 测试用例 T-01 ~ T-06

### T-01 · `/api/meetings/{id}/summary` endpoint 返回 含 `summary_json` 字段

- **前置**: P-1 ~ P-5 pass + `$MID` 保存
- **目标**: 校验 backend endpoint 已升级, 返回 schema 含 `summary_json` 顶级字段.
- **步骤**:
  ```bash
  curl -sS https://aimeeting.zhzjpt.cn/api/meetings/$MID/summary \
    -H "Authorization: Bearer $TOKEN" | jq 'keys'
  ```
- **Pass 条件**:
  - 输出 包含 字符串 `"summary_json"` (jq keys 列表)
  - 同时 含 `"summary_md"` + `"status"`
- **Fail 条件**:
  - 缺 `"summary_json"` key → endpoint 未升级 (commit 没生效)
  - HTTP 5xx → 后端 crash
- **必须复述**: jq keys 完整 输出 (JSON array)

### T-02 · 已 summary ready 的老会议 仍 work (backward compat)

- **前置**: T-01 pass
- **目标**: 老会议 (在 commit `4b6add2` 之前 跑过的 summary) `summary_json` 应该是 null,
  但 `summary_md` 仍 返回 老 markdown, `status: "ready"`.
- **步骤**:
  ```bash
  curl -sS https://aimeeting.zhzjpt.cn/api/meetings/$MID/summary \
    -H "Authorization: Bearer $TOKEN" | \
    jq '{ status, summary_md_len: (.summary_md | length), summary_json_type: (.summary_json | type) }'
  ```
- **Pass 条件**:
  - `status == "ready"`
  - `summary_md_len > 0`
  - `summary_json_type == "null"` OR `summary_json_type == "object"`
    - 老会议: `"null"` (没regenerate 过)
    - 新会议: `"object"` (commit 4b6add2 之后 跑的, 直接有)
- **Fail 条件**:
  - status != ready → 这个 meeting 还没生成 summary, P-4 选 错 了 (BLOCKED 不计)
  - summary_md_len == 0 → 回归 (老 summary 丢了)
- **必须复述**: jq 输出原文

### T-03 · 强制 regenerate 后 `summary_json` 是 object 含 完整 schema

- **前置**: T-02 pass
- **目标**: 触发 regenerate → 等 LLM 跑完 → 校验 `summary_json` 含 完整 schema.
- **步骤**:
  1. ```bash
     curl -sS -X POST https://aimeeting.zhzjpt.cn/api/meetings/$MID/summary/regenerate \
       -H "Authorization: Bearer $TOKEN" | jq .
     ```
     **Pass**: 返回 `{ "summary_md": null, "status": "pending" }`
  2. 等 60-120 秒 (LLM 跑) — 用 sleep 或 poll
  3. ```bash
     curl -sS https://aimeeting.zhzjpt.cn/api/meetings/$MID/summary \
       -H "Authorization: Bearer $TOKEN" | jq '.summary_json | { has_title: (.title != null), has_topics: (.topics != null), n_topics: (.topics | length), n_risks: (.risks | length), n_next_steps: (.next_steps | length) }'
     ```
- **Pass 条件**:
  - `has_title == true`
  - `has_topics == true`
  - `n_topics >= 0` (整数, 可以是 0)
  - `n_risks >= 0`
  - `n_next_steps >= 0`
- **Fail 条件**:
  - 上述 任一 字段 缺 → schema 不完整
  - 60s 后 status 仍 pending → LLM hung (不算 fail, 重试 直到 120s; 仍 pending → 算 fail)
- **必须复述**: 第 1 / 第 3 步 完整 jq 输出 + 等待时间 (秒数)

### T-04 · summary_json.topics[*].speakers 含 AI 专家 (不再 剔除)

- **前置**: T-03 pass + `$MID` 是 hybrid 或 auto 模式 (有 AI 发言)
- **目标**: 痛点 3 核心 验证 — AI 专家 发言 出现在 summary 的 speaker 列表里.
- **步骤**:
  ```bash
  curl -sS https://aimeeting.zhzjpt.cn/api/meetings/$MID/summary \
    -H "Authorization: Bearer $TOKEN" | \
    jq '[.summary_json.topics[].speakers[] | select(.speaker_type == "ai") | .speaker_name] | { ai_speaker_count: length, ai_speakers: . }'
  ```
- **Pass 条件**:
  - 若 该 meeting 有 AI 参与: `ai_speaker_count > 0`, `ai_speakers` 含 AI 名字
  - 若 该 meeting 没 AI 参与 (mode=human): `ai_speaker_count == 0` 合理
- **Fail 条件**:
  - meeting 是 hybrid/auto 但 `ai_speaker_count == 0` — AI 没被纳入 summary
    (这是 痛点 3 没修, RED)
- **必须复述**: jq 输出 + meeting mode (取自 `/api/m/meetings/$MID` 的 `mode` 字段)

### T-05 · summary_json.topics[*].action_items[*].source_line_id 非 null (任务溯源)

- **前置**: T-03 pass
- **目标**: 验证 action_items 链回 transcript line.
- **步骤**:
  ```bash
  curl -sS https://aimeeting.zhzjpt.cn/api/meetings/$MID/summary \
    -H "Authorization: Bearer $TOKEN" | \
    jq '[.summary_json.topics[].action_items[] | { has_source: (.source_line_id != null), text: .text[0:30] }]'
  ```
- **Pass 条件 A** (有 action_items): 至少 1 个 `has_source == true`
- **Pass 条件 B** (无 action_items): 输出 `[]` — meeting 没抽出待办 — 不算 fail
- **Fail 条件**:
  - 多个 action_items 全都 `has_source == false` → LLM 没回填 line_id, 任务溯源 不会 work
- **必须复述**: jq 输出 完整 array

### T-06 · UI smoke — Mobile summary 页 渲染 结构化 view (无 报错)

- **前置**: T-03 pass + meeting URL
- **目标**: 浏览器 打开 `/m/meetings/$MID/summary`, 看到 topic + stance + 任务溯源 chip,
  控制台 没 红 error.
- **步骤**:
  1. 浏览器 (推荐 Chrome) 打开 <https://aimeeting.zhzjpt.cn> 用 `demo.lijg@futian.gov.cn` / `demo123` 登录
  2. 跳 `https://aimeeting.zhzjpt.cn/m/meetings/<MID>/summary`
  3. 等纪要 ready (若还在 generating 等 1-2 min)
  4. DevTools (F12) 开 Console
  5. 滚到底, 截图整页
- **Pass 条件**:
  - 页面 渲染 **不是** 单 一篇 markdown 文字 — 而是 **多 张 topic 卡片**, 每个 含:
    - topic 名 + summary
    - speaker 列表 (带 颜色 pill: "支持" 绿 / "顾虑" 黄 / "反对" 红 / "中立" 灰)
    - decision 绿色 块
    - action_items 卡 (含 复选框 + 任务溯源 chip "跳实录")
  - 至少 1 个 任务溯源 chip 字面 显示 "跳实录" 文字
  - Console **无** 红色 error log (warning 可接受)
- **Fail 条件**:
  - 页面 只显 markdown / 不显 结构化 → MSummaryV2 没生效 (前端 wire 错)
  - 任务溯源 chip 字面 缺 "跳实录" → SourceChip 没渲染
  - Console 有 `Cannot read property '...' of undefined` / `Failed to ...` → 数据漏校
- **必须复述**: 整页 截图 + Console 截图 + 至少 1 个 topic 卡的 完整 文字 内容 (复述 ≥ 5 行)

## 风险 + PM 后置 manual 测试 提示

- **风险 1** (LLM JSON 出错): LLM 偶尔 输出 不合法 JSON, fallback 走 老 markdown.
  → PM 观察: regen 几场 老 会议 看 `summary_json` 是否 是 object (而非 null).
  若 几次 都 null → JSON prompt 需 进一步 调优.
- **风险 2** (source_line_id LLM 没回填): LLM 没看到 [L:<id>] 标签 或 忘了 填.
  → PM 观察: T-05 多场会议 跑 看 has_source 命中率.
- **风险 3** (Mobile vs Web focus 格式不一致): Mobile 用 `?focus=user-<id>` (前缀),
  Web 用 `?focus=<id>` (plain). 实际 设计 是 双端 各自 规约. 跨平台分享 link 可能 不直接 work.
