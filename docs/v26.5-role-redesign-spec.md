## 一、三层组织建模

```
组织层级        系统抽象        典型 role
─────────────────────────────────────────────
局 / 单位     ↔  Workspace    ↔  owner / admin / leader
部门 / 科室    ↔  User         ↔  manager
个体 AI 专家   ↔  Agent        ↔  (Agent.primary_user_id 指 manager)
```

### 典型场景:福田住建局

```
Workspace: 福田住建局
├── owner:     bluesurfiregpt@gmail.com    (乙方代建 / 签合同时)
├── admin:     局长 李建国                  (一把手,全权)
├── leader:    副局长 张明                  (能读能写,等同 admin)
├── manager × N:  科长 们
│   ├── 房屋安全管理科长(管 房屋安全 AI / 房屋政策 AI / 安全巡查 AI)
│   ├── 物业监管科长(管 物业监管 AI)
│   └── ...
└── member × N:  普通员工(看 + 邀请 AI,不管理)
```

## 二、最终 5 角色矩阵

| Role | 中文 | 全局写 | 全局读 | 个人 AI 管理 | 任免他人 |
|---|---|---|---|---|---|
| **owner** | 工作空间拥有者 | ✓ | ✓ | ✓(若被指定为 primary_user) | ✓ 任免任何角色 |
| **admin** | 局长 / 一把手 | ✓ | ✓ | ✓(若被指定为 primary_user) | ✓ 任免 leader/manager/member(**不可改 owner**) |
| **leader** | 副局长 / 等同 admin 别名 | ✓ | ✓ | ✓ | ✓ 任免 manager/member(**不可改 admin/owner**) |
| **manager** | 科长 / 部门 AI 维护人 | 限范围 ✓(他 primary 的 AI) | ✓ | ✓ | ❌ |
| **member** | 普通员工 | ❌ | 限范围 | ❌ | ❌ |

**关键规则**:
1. `Agent.primary_user_id` 推荐指向 `owner`/`admin`/`leader`/`manager`(不指向 `member`)
2. AI 管理范围(KB / 长期记忆 / Agent 配置 / 任务沉淀)= `caller is agent.primary_user` OR `caller role in (owner/admin/leader)`
3. `manager` 不需要绑死一个 AI,管几个 AI 由 `Agent.primary_user_id = manager.id` 反向查
4. v21 `WorkspaceMembership.bound_agent_id` 字段 **作废**(保留向后兼容,不再校验)

## 四、ABAC 规则(具体到端点)

### 4.1 Workspace 级写权限 → `require_admin_or_owner`(改名)

需要 `admin / leader / owner`:
- 邀请新成员 / 改成员角色(`/api/team/*` 部分写)
- 创建会议(`/api/meetings POST`)
- 派任务(`/api/me/tasks/*/dispatch`)
- v26.3 召集人模式控制(`/api/meetings/*/orchestrate/*`)
- v26.3-07 裁决分歧(`/api/meetings/*/consensus/*/review`)

### 4.2 Agent 级写权限 → `require_agent_manager`(新 helper)

需要 `caller is agent.primary_user_id` OR `caller is admin/leader/owner`:
- 修改 agent 配置(`PATCH /api/agents/{id}`)
- 删除 agent(`DELETE /api/agents/{id}`)
- 上传 KB 文档(`POST /api/knowledge-bases/{kb_id}/documents`,**前提 KB.owner_agent_id 指向某 agent**;无 owner 的 KB 退到 require_admin)
- 删 KB 文档(`DELETE /api/knowledge-bases/{kb_id}/documents/{doc_id}`)
- 写长期记忆(`POST /api/memory`)
- 审批 任务沉淀 KB(v26.2 chain 加 gate)

### 4.3 Workspace 级读权限 → 不动(workspace 隔离)

- 看 agent 列表 / 会议列表 / KB 列表 / 任务列表 — 所有 workspace 成员都能看
- 唯一例外:member 看 task 限 `assignee_user_id == me OR assignee_agent.primary_user_id == me`(已实现)

### 4.4 Platform Admin 跨 ws(v26.4 已实现,不动)

`bluesurfiregpt@gmail.com` 切到任何 workspace 视为 owner,所有写权限自动通过。

## 八、风险 + 决策点

### ⚠️ 风险 1:expert role 数据迁移 影响 现有 v25 demo seed

`backend/app/demo_seed.py:259` 创建 10 个 user `role='expert'` + `bound_agent_name='物业监管AI专家'`。
- 迁移后这些 user 变 `role='manager'`,`bound_agent_id` 保留但失效
- v25 demo 测试用例 如果 hard-code 检查 `role='expert'` 会 fail
- 处置:demo_seed.py 同步改成 `role='manager'`,Kimi 用例同步

### ⚠️ 风险 2:`is_leader_or_admin` 别名迁移

代码库 ~30 处用了这 helper,改名要全 grep replace。保险做法:加 `is_workspace_admin = is_leader_or_admin` 别名,旧名 deprecated 1 个里程碑。

### ⚠️ 决策点 1:member 能不能 创建会议?

当前 hybrid 会议任何 workspace 成员都能创建。新 ABAC 下,member 应该:
- A:仍能创建 hybrid 会议(轻量协作)
- B:不能创建,需要 manager+ 才能开会

我推 **A**(轻量保留 — 普通员工偶尔自己开会议没坏处)。

### ⚠️ 决策点 2:`KnowledgeBase.owner_agent_id` 必填吗?

- 必填:每个 KB 必须归一个 agent 管,但 cross-agent KB 不好表达
- 可空:nullable,无 owner 的 KB 退到 admin-only 写

我推 **可空**(灵活,默认行为)。
